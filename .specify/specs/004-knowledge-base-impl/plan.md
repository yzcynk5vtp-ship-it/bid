# Implementation Plan: 知识库 (Knowledge Base)

**Branch**: `004-knowledge-base-impl` | **Date**: 2026-05-20 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `/specs/004-knowledge-base-impl/spec.md`

## Summary

知识库模块作为投标的核心枢纽，包含了三个核心能力：方案管理（项目档案台账与 AI 案例库）、资质证书管理（集中录入、过期告警、借用审计）和保证金跟踪管理。
技术方案采用前后端分离架构，严格遵循 **FP-Java Profile**：
1. **后端**：在 `com.xiyu.bid` 下扩展/重构 `casework`、`businessqualification` 和 `qualification` 包。所有的业务校验、到期时间计算、复用计数规则写入不可变的 `record` 和纯核心类（Pure Core）。Repository、Service 仅进行用例编排和持久化（Imperative Shell）。涉及项目范围接口全部由 `ProjectAccessScopeService` 保护，通过 `Flyway` 管理数据库迁移。
2. **前端**：采用 Vue 3 composition API 对 `src/views/Knowledge/` 目录进行组件化细粒度重构，补充 Popover 文件分类浮层、双轴日期选择器和 Drawer 的交互。
3. **AI 切片**：项目结项触发异步线程调用大模型（对接已有的文档分析/侧车端），把标书原文切片并写入 `KnowledgeCase` 表。

## Technical Context

**Language/Version**: Java 21 (Backend), Node.js v20 / Vue 3 / Vite 5 (Frontend)

**Primary Dependencies**: Spring Boot 3.3, JPA (Hibernate), Element Plus, Pinia, Axios, ArchUnit (Architecture Tests)

**Storage**: MySQL 8.0 (via Flyway migrations), Redis 6.2 (token metadata/locks)

**Testing**: JUnit 5 (Unit Tests), ArchUnit (Architecture rules check), Playwright (E2E testing)

**Target Platform**: Linux (Docker containerized target), macOS (local development)

**Project Type**: Web application (frontend + backend)

**Performance Goals**: UI rendering < 200ms, ZIP file pack/export completed under 5 seconds, AI async extraction < 5 mins

**Constraints**: Strict FP-Java constraint (Pure Core vs Imperative Shell), 300 lines limit per file, mandatory Project Access Guard

**Scale/Scope**: System-wide knowledge hub supporting 10k users, ~10 relational tables, 5 main visual panels

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

| Principle / Rule | Compliance Status | Implementation Strategy |
|------------------|-------------------|-------------------------|
| **FP-Java Profile** | ✅ Compliant | Core domain logic (e.g. expiration date comparisons, validation, audit log generation) must only use immutable Records/classes. Imperative shells handle JPA and Controller orchestration. No void methods in Core. |
| **Real-API Only** | ✅ Compliant | No mock routing. All UI interactions use actual Axios requests to backend routes. E2E tests target live Spring Boot dev backend. |
| **Test-Driven Development (TDD)** | ✅ Compliant | Introduce test classes checking core domain functionality (e.g. `QualificationDomainTest`) before finalizing services. Run `ArchitectureTest` to enforce layer boundaries. |
| **Split-First & Simplicity** | ✅ Compliant | Disallow god-classes. Service classes will delegate to specific Policy and Helper files. Keep Java files < 200 lines (hard limit at 300). |
| **Project Access Guard** | ✅ Compliant | All REST API queries referencing projects check permission via `ProjectAccessScopeService` to isolate data. |
| **Flyway Migrations** | ✅ Compliant | Any updates to `qualification`, `casework`, `case_table` tables use versioned `.sql` scripts inside `backend/src/main/resources/db/migration-mysql/`. |
