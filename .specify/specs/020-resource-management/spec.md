**Feature Name**: 资源管理 — 保证金管理与招标平台账号管理
**Feature Branch**: `agent/qoder/resource-management`
**Created**: 2026-06-09
**Status**: Draft
**Input**: User description: "建设投标项目保证金统一台账，集中展示缴纳、退还、超期情况；建设招标平台账号统一管理模块，实现平台台账、CA统一管理、领用审批流、到期提醒、审计可追溯。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 保证金台账查看与筛选 (Priority: P1)

投标管理员希望在一个统一页面查看所有项目的保证金缴纳、退还、超期情况，支持按项目编号、项目名称、招标主体、缴纳状态、退还状态、缴纳时间范围等多维度筛选和搜索，并支持导出数据。

**Why this priority**: 保证金资金占用与回收进度是投标业务的核心财务指标，当前系统缺乏统一视图，业务负责人需要手动汇总多个项目数据，效率低下且容易遗漏超期未退保证金。

**Independent Test**: 以投标管理员身份登录 → 进入「资源管理」→「费用管理」→「保证金退还跟踪」→ 页面展示保证金台账列表 → 使用搜索区筛选条件（如缴纳状态=已缴纳）→ 列表正确过滤 → 点击「导出」按钮 → 下载 Excel 文件内容正确。

**Acceptance Scenarios**:

1. **Given** 系统存在多个项目的保证金记录，**When** 投标管理员访问保证金台账页面，**Then** 页面展示所有可见项目的保证金信息（项目编号、项目名称、招标主体、项目负责人、投标负责人、保证金金额、缴纳状态、缴纳时间、退回金额、转服务费金额、退回状态、应退日期、超期天数）
2. **Given** 保证金台账页面，**When** 用户输入项目编号/名称关键词并搜索，**Then** 列表只展示匹配的项目
3. **Given** 保证金台账页面，**When** 用户选择缴纳状态（已缴纳/未缴纳/部分缴纳）、退还状态（已退/未退/部分退/转服务费）、缴纳时间范围等筛选条件，**Then** 列表按组合条件过滤
4. **Given** 保证金台账页面，**When** 用户点击「导出」按钮，**Then** 系统生成并下载包含当前筛选结果的 Excel 文件
5. **Given** 保证金台账页面，**When** 用户点击某行「项目名称」，**Then** 跳转至对应项目详情页

### User Story 2 - 保证金统计卡片 (Priority: P1)

业务负责人希望在保证金台账页面顶部看到关键指标的汇总卡片，快速掌握整体资金状况。

**Why this priority**: 统计卡片提供一目了然的关键指标，帮助管理层快速决策，无需逐行查看列表即可发现超期风险。

**Acceptance Scenarios**:

1. **Given** 保证金台账页面，**Then** 页面顶部展示 5 张统计卡片：保证金支付总额、保证金未退回总额、保证金未退回笔数、超期未退回金额、超期未退回笔数
2. **Given** 用户应用筛选条件后，**When** 筛选结果变化，**Then** 统计卡片数据同步更新为当前筛选范围内的汇总值

---

### User Story 3 - 招标平台账号台账 (Priority: P1)

投标管理员希望集中维护所有投标平台的账号信息，包括平台名称、URL、登录账号、密码、绑定联系人等。

**Why this priority**: 当前平台账号信息分散在各处，人员变动时交接困难，密码遗忘频繁，需要统一台账管理。

**Acceptance Scenarios**:

1. **Given** 招标平台账号管理页面，**When** 投标管理员查看列表，**Then** 展示所有平台账号（平台名称、平台URL、登录账号、绑定联系人、是否有CA、创建时间）
2. **Given** 招标平台账号管理页面，**When** 投标管理员/组长/专员点击「新增」，**Then** 弹出表单，填写平台名称、URL、账号、密码、绑定联系人后保存成功
3. **Given** 招标平台账号管理页面，**When** 投标管理员/组长点击「编辑」，**Then** 可修改平台账号信息
4. **Given** 招标平台账号管理页面，**When** 投标管理员/组长点击「删除」，**Then** 弹出确认后删除记录（软删除）
5. **Given** 招标平台账号管理页面，**When** 项目负责人访问，**Then** 仅可见平台名称、URL、是否有CA三个字段，不可见账号密码

---

### User Story 4 - CA 统一管理 (Priority: P1)

投标管理员希望统一登记每张CA证书的信息，包括CA类型、印章信息、密码、有效期、保管人等。

**Why this priority**: CA证书是投标的关键凭证，有效期管理和保管人追踪直接影响投标成功率，需要系统化管控。

**Acceptance Scenarios**:

1. **Given** CA管理页面，**When** 用户查看列表，**Then** 展示所有CA记录（CA编号、CA类型、所属平台、印章信息、有效期、保管人、状态）
2. **Given** CA管理页面，**When** 投标管理员/组长/专员点击「新增CA」，**Then** 填写CA类型、所属平台、印章信息、密码、有效期、保管人后保存
3. **Given** CA管理页面，**When** 投标管理员/组长编辑CA信息，**Then** 可修改除CA编号外的所有字段
4. **Given** CA管理页面，**When** 投标管理员/组长删除CA，**Then** 确认后软删除
5. **Given** CA管理页面，**When** 投标专员查看，**Then** 仅可见自己保管的CA的完整信息，其他CA部分脱敏

### User Story 5 - 账号/CA 领用审批流 (Priority: P2)

项目人员需要借用平台账号或CA时，可发起领用申请，经保管员审批后线下交付，使用完毕后登记归还。

**Why this priority**: 领用审批流确保账号/CA的借用可追溯，防止丢失或滥用，满足审计要求。

**Acceptance Scenarios**:

1. **Given** 账号/CA领用页面，**When** 任何角色用户点击「发起领用」，**Then** 选择领用类型（账号/CA）、选择具体账号/CA、填写领用事由、预计归还时间后提交申请
2. **Given** 领用申请已提交，**When** 保管员（投标管理员/组长/CA保管人）查看审批列表，**Then** 可见待审批的申请
3. **Given** 待审批的领用申请，**When** 保管员点击「审批通过」，**Then** 申请状态变为「已批准」，账号/CA状态变为「已借出」，记录借出时间和审批人
4. **Given** 已借出的账号/CA，**When** 借用人或保管员点击「登记归还」，**Then** 填写实际归还时间后，账号/CA状态变为「在库」，申请状态变为「已归还」
5. **Given** 领用申请列表，**When** 任何用户查看，**Then** 只可见自己发起或需要自己审批的申请（管理员/组长可见全部）

---

### User Story 6 - 到期提醒机制 (Priority: P2)

系统自动在CA有效期前1个月、借用归还前1个月发送提醒，避免CA过期影响投标或借用超期。

**Why this priority**: 自动提醒减少人工检查成本，避免因CA过期导致无法投标的重大损失。

**Acceptance Scenarios**:

1. **Given** 系统存在有效期在30天内的CA，**When** 系统每日扫描，**Then** 向CA保管人、投标管理员、组长发送CA即将过期提醒
2. **Given** 系统存在预计归还日期在30天内的借出记录，**When** 系统每日扫描，**Then** 向借用人、保管人、投标管理员、组长发送即将到期归还提醒
3. **Given** CA已过期或借用已超期，**When** 系统每日扫描，**Then** 发送超期告警（标红/加急）给相关人员
4. **Given** 提醒通知页面，**When** 投标管理员查看，**Then** 可配置全局告警规则（提前提醒天数、提醒方式等）

---

### User Story 7 - 操作日志与审计 (Priority: P2)

所有平台账号和CA的信息变更、借出归还记录均需记录操作日志，支持审计追溯。

**Why this priority**: 审计可追溯是合规要求，也是问题排查的重要依据。

**Acceptance Scenarios**:

1. **Given** 平台账号或CA发生新增/编辑/删除操作，**When** 操作完成，**Then** 系统自动记录操作日志（操作人、操作时间、操作类型、变更前/后内容）
2. **Given** 账号/CA发生借出/归还操作，**When** 操作完成，**Then** 系统记录借还日志（借用人、审批人、借出时间、归还时间）
3. **Given** 操作日志页面，**When** 投标管理员/组长查看，**Then** 可按时间范围、操作类型、操作人筛选日志
4. **Given** 操作日志页面，**When** 投标专员查看，**Then** 仅可见自己保管的CA相关的操作日志

## Edge Cases

- **保证金数据同步延迟**：项目立项时缴纳信息已确定，但退回金额和转服务费金额需等项目结项后才同步，列表应支持展示「待结项」状态的空白值
- **多项目同一平台**：不同项目可能使用同一招标平台的不同账号，平台台账与项目之间是多对多关系
- **CA密码安全**：CA密码和平台账号密码需加密存储，前端展示时脱敏，仅授权人员可查看明文
- **领用冲突**：同一CA同一时间只能被一人借出，系统需防止重复审批通过
- **批量导入**：支持从Excel批量导入平台账号和CA信息，导入时校验数据格式和必填项
- **权限边界**：项目负责人仅可见部分字段，不能操作增删改，不能查看明文密码（除非自己的借用申请被批准）

## Requirements *(mandatory)*

### Functional Requirements

#### 保证金管理模块

- **FR-001**: 系统 MUST 提供保证金台账列表接口，支持分页、排序、多条件筛选（项目编号、项目名称、招标主体、缴纳状态、退还状态、缴纳时间范围）
- **FR-002**: 系统 MUST 提供保证金统计接口，返回支付总额、未退回总额、未退回笔数、超期未退回金额、超期未退回笔数
- **FR-003**: 系统 MUST 支持保证金数据导出为 Excel，导出内容包含当前筛选结果的全部字段
- **FR-004**: 保证金缴纳金额、缴纳状态 MUST 从项目立项阶段信息自动同步
- **FR-005**: 保证金退回金额、转服务费金额、退回状态 MUST 从项目结项环节自动同步
- **FR-006**: 关联项目信息（招标主体、项目负责人、投标负责人）MUST 从项目模块自动带出
- **FR-007**: 超期计算 MUST 以「应退日期」为基准，超期天数 = 当前日期 - 应退日期（应退日期为空则不计算）

#### 招标平台账号管理模块

- **FR-008**: 系统 MUST 提供招标平台账号的增删改查接口，支持分页列表、按平台名称搜索
- **FR-009**: 平台账号密码 MUST 加密存储（AES-256），前端默认脱敏展示，授权人员可查看明文
- **FR-010**: 系统 MUST 提供CA证书的增删改查接口，CA信息与平台账号关联
- **FR-011**: CA密码 MUST 加密存储，前端默认脱敏展示
- **FR-012**: 系统 MUST 提供账号/CA领用申请的发起、审批、归还全流程接口
- **FR-013**: 领用申请 MUST 支持工作流状态：待审批 → 已批准 → 已借出 → 已归还 / 已驳回
- **FR-014**: 系统 MUST 实现定时任务，每日扫描CA有效期和借用归还日期，发送提醒通知
- **FR-015**: 系统 MUST 记录所有平台账号和CA的操作日志（新增、编辑、删除、借出、归还）
- **FR-016**: 系统 MUST 支持招标平台账号和CA信息的批量导入（Excel模板）

### Non-Functional Requirements

- **NFR-001**: 保证金台账列表查询响应时间 < 500ms（单页50条）
- **NFR-002**: 统计卡片数据与列表筛选结果实时同步
- **NFR-003**: 密码加密存储，密钥由系统统一管理，不得硬编码
- **NFR-004**: 操作日志保留时间 ≥ 2年

## Key Entities *(include if feature involves data)*

- **ProjectDeposit**（保证金记录）：项目保证金台账实体，关联 Project，记录缴纳金额、缴纳状态、缴纳时间、退回金额、转服务费金额、退回状态、应退日期等
- **BidPlatform**（招标平台）：招标平台基础信息实体，记录平台名称、URL、登录账号、密码（加密）、绑定联系人等
- **PlatformCA**（CA证书）：CA证书实体，关联 BidPlatform，记录CA类型、印章信息、密码（加密）、有效期、保管人、状态等
- **CaBorrowRecord**（CA/账号领用记录）：领用申请实体，记录领用类型、关联CA/账号、借用人、审批人、借出时间、预计归还时间、实际归还时间、状态等
- **PlatformOperationLog**（操作日志）：平台账号和CA的操作审计日志，记录操作人、操作时间、操作类型、变更内容等
- **Project**（项目）：已有实体，保证金数据的数据源

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 保证金台账页面可正确展示所有项目的保证金信息，筛选、搜索、导出功能正常
- **SC-002**: 统计卡片数据与列表筛选条件联动，数值准确
- **SC-003**: 招标平台账号和CA的增删改查功能正常，权限控制符合角色矩阵
- **SC-004**: 账号/CA领用审批流完整跑通：发起 → 审批 → 借出 → 归还
- **SC-005**: 到期提醒任务每日执行，CA过期前30天、归还前30天正确发送提醒
- **SC-006**: 操作日志完整记录所有变更，支持筛选查询
- **SC-007**: 单元测试覆盖率 ≥ 80%（domain/core 层）
- **SC-008**: ArchUnit 门禁通过（`FPJavaArchitectureTest`、`MaintainabilityArchitectureTest`、`ProjectAccessGuardCoverageTest`）
- **SC-009**: Playwright E2E 测试覆盖保证金台账页面和平台账号管理页面的核心流程

## Assumptions

- 项目立项和结项模块已提供保证金相关数据的读写接口
- 用户角色体系（投标管理员、投标组长、投标专员、项目负责人）已在系统中定义
- 通知系统（站内信/邮件）已具备基础能力，本功能只需调用发送接口
- 密码加密使用系统已有的加密工具类
