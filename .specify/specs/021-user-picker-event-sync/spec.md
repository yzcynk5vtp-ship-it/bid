# Feature Specification: 选人控件统一 + 事件库同步启用

**Feature Branch**: `agent/mimo/user-picker-event-sync`

**Created**: 2026-06-24

**Status**: Draft

**Input**: User description: "现在登录的问题已经解决了 直接走OSS鉴权就可以 但是，现在系统中有很多选人的控件，你看一下，这个还是要走事件库 走订阅"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 选人控件统一为本地数据源 (Priority: P1)

系统中所有需要选择人员的场景（任务执行人、标讯指派、项目负责人、审核人、保管员、@提及、协作成员等共 22 处）应统一使用同一个选人组件，该组件背后查询的是本地已同步的用户数据，而非每次输入都触发远程 OSS 接口调用。当前 22 处选人实现碎片化为 4 套模式（远程搜索、任务候选人、标讯候选人、空列表），需收敛为统一组件 + 统一数据源。

**Why this priority**: 选人控件是日常高频操作入口，碎片化导致行为/样式/数据格式不一致，且 3 处控件因依赖未填充的 `userStore.users` 而下拉始终为空，属于功能性失效。统一后所有选人场景行为一致、数据准确。

**Independent Test**: 在任意一个业务页面（如任务表单、标讯指派对话框）中打开选人控件，输入关键字，能立即看到匹配的候选人列表，且列表数据来自本地已同步用户表。

**Acceptance Scenarios**:

1. **Given** 系统已通过事件库同步本地用户数据，**When** 用户在任务表单的"任务执行人"字段输入姓名关键字，**Then** 下拉显示匹配的启用状态用户，包含姓名、部门、角色信息
2. **Given** 用户在标讯指派对话框中，**When** 打开候选人列表，**Then** 显示按当前用户数据权限过滤后的候选人，与任务模块使用相同的选人组件
3. **Given** 用户在项目创建的"项目负责人"字段，**When** 打开下拉，**Then** 不再是空列表，而是显示可选的候选人（修复当前 `userStore.users` 失效问题）
4. **Given** 任意选人控件，**When** 选中一个用户后，**Then** 统一返回用户标识（userId）和显示信息（姓名+部门+角色），不再出现有的场景返回 `user.name` 字符串、有的返回 `userId` 的分裂

---

### User Story 2 - 事件库同步保持本地用户数据准实时 (Priority: P1)

启用事件库 SDK（Kafka 消费 OSS 组织架构变更事件），当 OSS 系统中用户/部门/岗位发生变更（新增、修改、离职、转岗）时，事件库自动消费并将变更同步到本地用户表，确保选人控件看到的数据与 OSS 系统保持准实时一致。当前事件库 SDK 已实现但默认关闭，本地用户数据可能过期或缺失。

**Why this priority**: 选人控件统一为本地数据源的前提是本地数据必须准确及时。若事件库未启用，本地用户表数据将逐渐与 OSS 系统脱节，选人控件看到的候选人可能已离职、转岗或信息过期。

**Independent Test**: 在 OSS 系统中修改某用户信息（如部门变更），等待事件库消费后，在本地选人控件中搜索该用户，能看到更新后的部门信息。

**Acceptance Scenarios**:

1. **Given** 事件库 SDK 已启用并连接 Kafka，**When** OSS 系统中某用户从 A 部门调转到 B 部门，**Then** 事件库消费变更事件后，本地用户表中该用户的 `department_code` 和 `department_name` 更新为 B 部门
2. **Given** 事件库 SDK 已启用，**When** OSS 系统中某用户离职（status 变为非在职），**Then** 本地用户表中该用户 `enabled` 标记为 false，选人控件不再展示该用户
3. **Given** 事件库 SDK 已启用，**When** OSS 系统新增一个投标系统用户，**Then** 事件库消费新增事件后，本地用户表新增该用户记录，选人控件可搜索到
4. **Given** 事件库 SDK 因 Kafka 连接异常暂时不可用，**When** 连接恢复后，**Then** 事件库自动恢复消费，期间错过的变更事件会被重新消费并同步

---

### User Story 3 - 候选人 API 统一为单一端点 (Priority: P2)

将当前分裂的两个候选人端点（`/api/tasks/assignment-candidates` 和 `/api/tenders/assignment-candidates`）合并为统一端点，通过上下文参数区分业务场景，后端按场景应用相应的权限过滤策略。统一后前端只需对接一个 API，减少分裂维护成本。

**Why this priority**: 候选人 API 分裂导致前端需要维护两套调用逻辑和两种返回结构，且标讯端点无权限过滤存在安全隐患。统一后降低维护成本并修复安全问题。

**Independent Test**: 调用统一后的候选人端点，分别传入任务和标讯上下文参数，验证返回的候选人列表按各自场景的权限规则正确过滤。

**Acceptance Scenarios**:

1. **Given** 当前用户具有全局数据权限（如管理员），**When** 调用统一候选人端点（任意上下文），**Then** 返回所有启用状态用户
2. **Given** 当前用户仅具有部门级数据权限，**When** 调用统一候选人端点并传入任务上下文，**Then** 仅返回当前用户可见部门内的候选人
3. **Given** 统一候选人端点，**When** 传入 `context=tender`，**Then** 应用标讯场景的权限过滤规则（当前标讯端点无过滤，统一后应补齐权限过滤）
4. **Given** 统一候选人端点，**When** 不传或传入无效 context 参数，**Then** 返回 400 错误并提示有效的 context 值

---

### User Story 4 - 选人组件统一封装 (Priority: P2)

封装一个可复用的选人组件，支持远程搜索模式（输入关键字异步查询）和预加载模式（一次性加载候选人列表），覆盖当前所有 22 处选人场景的使用需求。组件统一处理用户显示标签格式化、防抖搜索、空状态、加载状态等交互细节。

**Why this priority**: 没有统一组件封装，22 处选人逻辑各自实现，行为和样式不一致，后续维护每改一处都要改 22 次。统一组件后只需改一处。

**Independent Test**: 将任意一个现有选人场景替换为统一组件，验证功能行为与替换前一致或更好。

**Acceptance Scenarios**:

1. **Given** 统一选人组件，**When** 配置为远程搜索模式，**Then** 用户输入时触发异步查询，支持防抖，显示加载状态
2. **Given** 统一选人组件，**When** 配置为预加载模式，**Then** 组件挂载时一次性加载候选人列表，无需输入即可选择
3. **Given** 统一选人组件，**When** 选中某用户后，**Then** emit 的事件和数据格式统一（userId + 用户完整信息对象），父组件无需适配多种格式
4. **Given** 统一选人组件，**When** 候选人列表为空或搜索无结果，**Then** 显示友好的空状态提示

---

### Edge Cases

- 事件库 SDK 启动时 Kafka 不可用：应记录错误日志，不阻塞应用启动，本地用户表保留上次同步的数据继续可用
- 事件库消费到的事件对应的用户在 OSS 远程回查时已删除：应将本地用户标记为 `enabled=false`，选人控件不再展示
- 选人控件搜索关键字包含 SQL 特殊字符（%、_、\）：后端需正确转义，防止 LIKE 查询异常
- 事件库消费到重复事件（同一 eventKey）：需幂等处理，不重复写入
- 用户在选人控件选中某用户后，该用户随后被事件库标记为离职：已选中的值应保留但提交时后端校验该用户是否仍可用
- 本地用户表为空（事件库未同步过数据）：选人控件应显示空状态提示，而非报错

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 启用事件库 SDK，消费 OSS 组织架构变更事件（用户/部门/岗位），将变更准实时同步到本地用户表
- **FR-002**: 事件库 SDK 消费用户变更事件后 MUST 回查 OSS 目录接口获取最新用户详情，upsert 到本地 users 表，包含姓名、部门、角色、启用状态
- **FR-003**: 事件库 SDK 消费到用户离职事件（OSS 远程查无此用户）时 MUST 将本地用户标记为 `enabled=false`
- **FR-004**: 事件库 SDK 消费到重复事件时 MUST 幂等处理，不重复写入、不产生数据异常
- **FR-005**: 系统 MUST 提供统一的选人组件，支持远程搜索模式（输入关键字异步查询）和预加载模式（一次性加载列表）
- **FR-006**: 统一选人组件 MUST 替换当前所有 22 处碎片化选人实现，行为和样式保持一致
- **FR-007**: 统一选人组件 MUST 修复当前 3 处因依赖 `userStore.users` 而下拉为空的失效控件
- **FR-008**: 统一选人组件选中后 MUST 返回统一的数据格式（用户 ID + 用户完整信息对象），不再出现有的返回 `user.name` 字符串、有的返回 `userId` 的分裂
- **FR-009**: 系统 MUST 将 `/api/tasks/assignment-candidates` 和 `/api/tenders/assignment-candidates` 合并为统一端点，通过上下文参数区分业务场景
- **FR-010**: 统一候选人端点 MUST 对所有上下文场景应用数据权限过滤，修复当前标讯端点无权限过滤的安全隐患
- **FR-011**: 统一候选人端点 MUST 返回统一的字段结构，包含 userId、姓名、工号、角色码、角色名、部门码、部门名、启用状态
- **FR-012**: 选人控件查询本地用户数据时 MUST 仅返回 `enabled=true` 的用户
- **FR-013**: 选人控件搜索 MUST 支持按姓名和用户名模糊匹配，大小写不敏感
- **FR-014**: 事件库 SDK 启用后 MUST 不阻塞应用启动，Kafka 连接失败时降级为使用本地已有数据

### Key Entities *(include if feature involves data)*

- **本地用户（User）**：已从 OSS 同步到本地的用户记录，包含姓名、用户名、部门、角色、启用状态、外部组织用户 ID、最后同步时间。选人控件的核心数据源。
- **OSS 组织架构事件**：OSS 系统通过 Kafka 推送的用户/部门/岗位变更事件，包含事件类型（新增/修改/删除）、实体 ID、变更内容。事件库消费后触发本地同步。
- **选人组件**：可复用的前端组件，封装用户选择交互，支持远程搜索和预加载两种模式，统一数据格式和交互行为。
- **候选人上下文**：业务场景标识（如任务指派、标讯分配），决定统一候选人端点应用哪种权限过滤策略。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 所有 22 处选人场景使用统一的选人组件，无碎片化实现残留
- **SC-002**: 3 处原 `userStore.users` 失效控件（BasicInfoStep/TaskStep/CollaborationCenter）下拉列表正常显示候选人
- **SC-003**: 事件库 SDK 启用后，OSS 用户变更在 5 分钟内同步到本地用户表（Kafka 消费延迟 + 回查 OSS 耗时）
- **SC-004**: 选人控件搜索响应时间不超过 500 毫秒（本地数据库查询）
- **SC-005**: 候选人 API 合并为单一端点，前端不再维护两套调用逻辑和两种返回结构
- **SC-006**: 统一候选人端点对所有场景应用数据权限过滤，标讯场景不再返回无权限过滤的全部用户
- **SC-007**: 事件库 SDK Kafka 连接异常时不阻塞应用启动，选人控件使用本地已有数据继续可用
- **SC-008**: 选人控件选中后返回统一数据格式，前端无需适配多种格式

## Assumptions

- OSS 系统的 Kafka 事件总线已就绪，能够推送 BaseOssUser/BaseOssDept/BaseOssJob 三类事件
- 事件库 SDK 的 EHSY ClientSDK jar 已在 classpath 中（`@ConditionalOnClass` 已满足）
- 本地 users 表已通过历史同步或批量窗口同步初始化了基础数据，事件库启用后增量同步保持准实时
- 当前 `OrganizationDirectorySyncAppService` 的事件处理链路（回查 OSS + upsert 本地）已正确实现，启用 SDK 后可直接工作
- 前端 Element Plus 的 `el-select filterable remote` 能力满足统一选人组件的交互需求
- 统一候选人端点的权限过滤策略可复用现有 `ProjectAccessScopeService` 和 `RoleProfileService` 的能力
- 事件库 SDK 的 Kafka 配置（broker 地址、consumer group）由运维环境变量注入，不在本特性范围内配置
