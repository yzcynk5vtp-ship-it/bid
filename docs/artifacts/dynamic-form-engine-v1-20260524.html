表单自定义适配器升级方案

基于现有 WorkflowForm 配置基础设施，将其升级为全系统通用的"动态表单自定义引擎"，使标讯、项目管理、知识库、资源管理等所有表单页面均可由管理员配置字段，无需改代码即可调整。

📅 2026-05-24 ✍️ Cursor Agent 📁 docs/artifacts/ 🔄 v1.0

一、背景与目标

💡

核心问题

西域数智化投标管理平台现有表单均为硬编码，每次调整字段都要改代码——效率低、风险高、无法产品化。随着业务扩展，标讯、项目、知识库、资源等模块的表单需求差异化越来越大，亟需一套通用自定义引擎。

🚀

效率提升

管理员可在设计器中调整字段，无需开发介入，减少 80%+ 的表单变更工时

🔒

降低风险

配置变更不影响代码基线，每次变更均有版本记录，支持一键回滚

📦

产品化基础

不同客户、不同组织可拥有独立表单模板，实现真正的 SaaS 多租户定制

✅

已有基础

系统已具备完整的 WorkflowForm 配置基础设施（数据模型、版本管理、Admin Designer、运行时加载接口），本方案基于此扩展复用，避免重复建设。

2.3 关键缺口

缺口 1：表单未与业务模块绑定 — 现有 WorkflowForm 仅服务于 OA 审批提交，未作为其他业务的动态字段来源
缺口 2：字段类型不完整 — 前端渲染器缺 SECTION / TABLE / PERSON / PROJECT / QUALIFICATION 等常用类型
缺口 3：字段级权限缺失 — 无法控制不同角色/组织看到不同字段
缺口 4：条件逻辑缺失 — 无法配置字段间依赖（如选"A类"时显示特定字段）
缺口 5：字段验证规则单一 — 仅支持 required 布尔值，不支持正则/范围/自定义消息
缺口 6：API 层未改造 — 各业务 Controller 仍直接接收硬编码 DTO，未从动态表单 schema 派生验证规则

5.1 新增表：form_definition_registry

CREATE TABLE form_definition_registry (
    id              BIGINT AUTO_INCREMENT PRIMARY KEY,
    scope           VARCHAR(64) NOT NULL COMMENT '业务域标识: tender.entry / project.basic / resource.expense / knowledge.case',
    scope_label     VARCHAR(128) NOT NULL COMMENT '业务域中文名',
    version         INT NOT NULL DEFAULT 1,
    schema_json     JSON NOT NULL COMMENT '字段定义JSON',
    enabled         BOOLEAN NOT NULL DEFAULT TRUE,
    org_id          BIGINT DEFAULT NULL COMMENT 'NULL=全局模板, 非NULL=租户级',
    created_by      VARCHAR(64) NOT NULL,
    created_at      DATETIME NOT NULL,
    updated_at      DATETIME NOT NULL,
    UNIQUE KEY uk_scope_org (scope, org_id)
);

5.2 新增表：form_field_visibility

CREATE TABLE form_field_visibility (
    id              BIGINT AUTO_INCREMENT PRIMARY KEY,
    definition_id   BIGINT NOT NULL,
    field_key       VARCHAR(64) NOT NULL,
    role_pattern    VARCHAR(64) DEFAULT NULL COMMENT '角色匹配: bid_specialist / manager / * (所有)',
    org_id          BIGINT DEFAULT NULL COMMENT 'NULL=所有组织',
    visible         BOOLEAN NOT NULL DEFAULT TRUE,
    readonly        BOOLEAN NOT NULL DEFAULT FALSE,
    hidden          BOOLEAN NOT NULL DEFAULT FALSE,
    CONSTRAINT fk_visibility_def FOREIGN KEY (definition_id) REFERENCES form_definition_registry(id),
    UNIQUE KEY uk_def_field_role (definition_id, field_key, role_pattern)
);

5.3 新增表：form_field_condition

CREATE TABLE form_field_condition (
    id              BIGINT AUTO_INCREMENT PRIMARY KEY,
    definition_id   BIGINT NOT NULL,
    source_field    VARCHAR(64) NOT NULL COMMENT '触发字段',
    operator        VARCHAR(32) NOT NULL COMMENT 'eq / neq / in / contains / gt / lt',
    target_value    VARCHAR(255) DEFAULT NULL COMMENT '目标值，多值用逗号分隔',
    action          VARCHAR(32) NOT NULL COMMENT 'show / hide / require / skip',
    target_field    VARCHAR(64) DEFAULT NULL COMMENT '受影响字段',
    CONSTRAINT fk_condition_def FOREIGN KEY (definition_id) REFERENCES form_definition_registry(id)
);

六、API 设计

6.1 Admin API（需 ADMIN 角色）

Method Path 说明
GET/api/admin/form-definitions列表（支持 scope 过滤、org_id 过滤）
POST/api/admin/form-definitions创建定义（含 schema_json）
PUT/api/admin/form-definitions/{id}更新定义
DELETE/api/admin/form-definitions/{id}删除（软删除，仅 ADMIN 可操作）
POST/api/admin/form-definitions/{id}/publish发布版本（写入 workflow_form_template_version + invalidate cache）
POST/api/admin/form-definitions/{id}/visibility批量保存字段可见性规则
POST/api/admin/form-definitions/{id}/conditions批量保存条件逻辑规则

6.2 Runtime API（需认证用户）

Method Path 说明
GET/api/form-definitions/{scope}/active获取运行时 schema（含权限过滤 + Redis 缓存）
POST/api/form-definitions/{scope}/validate纯校验接口（不保存，返回错误列表）
POST/api/form-definitions/{scope}/submit提交并触发业务逻辑（分发到各 scope handler）

6.3 业务逻辑分发（AdaptiveFormService.submit）

public void submit(String scope, Map<String, Object> formData, String operator) {
    FormDefinition def = repository.findActive(scope);
    // 1. 权限校验 (org_id + role)
    // 2. 从 schema 生成校验规则 + 校验
    // 3. 分发到对应业务 handler
    switch (scope) {
        case "tender.entry"      -> tenderHandler.process(formData);
        case "tender.evaluation" -> tenderEvalHandler.process(formData);
        case "project.basic"     -> projectHandler.process(formData);
        case "project.evaluation"-> projectEvalHandler.process(formData);
        case "resource.expense"  -> expenseHandler.process(formData);
        case "resource.ca"      -> caCertHandler.process(formData);
        case "knowledge.case"    -> caseHandler.process(formData);
        case "knowledge.qual"    -> qualificationHandler.process(formData);
        default                  -> throw new UnsupportedScopeException(scope);
    }
}
