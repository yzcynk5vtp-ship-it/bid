## 〇、前置基础发现（必须先理解，否则后续对照全错）

### F1. 文档角色 code 与代码角色 code 是两套命名，需映射

文档的 7 个 code 在代码中**一个都不存在**（全仓 grep 0 命中）。代码用的是下划线风格。映射关系：

| 文档角色 | 文档 code | 代码 code | 代码显示名 | 代码 dataScope |
|---|---|---|---|---|
| 投标系统管理员 | `bid-SystemAdmin` | `admin` | 管理员 | all |
| 投标管理员 | `bidAdmin` | `bid_admin` | 投标部门管理员 | all |
| 投标组长 | `bid-TeamLeader` | `bid_lead` | 投标组长 | all |
| 投标项目负责人 | `bid-projectLeader` | `sales` | 项目负责人 | self |
| 投标专员 | `bid-Team` | `bid_specialist` | 投标专员 | self |
| 跨部门协同人员 | `bid-otherDept` | `bid_other_dept` | 跨部门协同人员 | self |
| 行政人员 | `bid-administration` | `admin_staff` | 行政人员 | self |

代码额外有 4 个文档未定义的角色：`bid_senior`（投标主管，合并 bid_admin+bid_lead）、`auditor`（审计员）、`manager`（经理）、`staff`（员工）、`task_executor`（任务执行人，文档里是"任意角色"子身份非独立角色）。

来源：`RoleProfileCatalog.java:10-148`

### F2. "项目级子身份"在代码里不是一等公民

文档定义的 4 个项目级子身份（投标负责人/投标辅助人员/标书审核人/任务执行人）在代码里**没有统一的成员模型**，分散承载：
- 投标负责人/辅助 → `ProjectLeadAssignment.primaryLeadUserId/secondaryLeadUserId`（`project/entity/ProjectLeadAssignment.java`）
- 标书审核人 → `BidDocumentReviewEntity.reviewerId`（`project/entity/BidDocumentReviewEntity.java`），临时选定
- 任务执行人 → `Task.assigneeId`

**关键后果**：后端绝大多数权限判定是**角色级（roleCode）**，不是**项目子身份级**。文档里大量"仅自己负责项目""仅分配给自己"的范围约束，后端要么不做，要么退化成 `ProjectAccessScopeService` 的"能访问该项目"（参与的即可），比文档窄约束宽。

### F3. legacy 兼容层会放大放行面（计算有效权限的钥匙）

`UserDetailsServiceImpl.authoritiesFor`（`auth/UserDetailsServiceImpl.java:46-112`）+ `RoleProfileCatalog.securityCompatLegacyRole`（`:198-205`）给每个新角色额外颁发旧 `ROLE_ADMIN/MANAGER/STAFF`：

| 代码角色 | 额外颁发的 legacy ROLE | 含义 |
|---|---|---|
| admin | ROLE_ADMIN | — |
| bid_admin | **ROLE_ADMIN** | 任何 `hasRole('ADMIN')` / `hasAnyRole(...'ADMIN'...)` 对 bid_admin 放行 |
| bid_lead | ROLE_MANAGER | 任何 `'MANAGER'` 注解放行 |
| bid_senior | ROLE_MANAGER | 同上 |
| sales | ROLE_MANAGER | 同上 |
| manager | ROLE_MANAGER | — |
| bid_specialist | ROLE_STAFF | 任何 `'STAFF'` 注解放行 |
| task_executor | ROLE_STAFF | 同上 |
| admin_staff | ROLE_STAFF | 同上 |
| auditor | ROLE_STAFF | 同上（默认分支） |
| bid_other_dept | **（无，跳过）** | 不发任何 legacy ROLE（2026-06-16 修复，见 memory） |

**因此常用注解的"有效放行角色"**：

| 注解字面 | 实际放行的代码角色 |
|---|---|
| `hasRole('ADMIN')` | admin, bid_admin |
| `hasAnyRole('ADMIN','MANAGER')` | admin, bid_admin, manager, bid_lead, bid_senior, sales |
| `hasAnyRole('ADMIN','MANAGER','STAFF')` | 几乎全员（除 bid_other_dept） |
| `hasAnyRole('ADMIN','BID_LEAD','BID_SENIOR')` | admin, bid_admin, bid_lead, bid_senior |
| `hasAnyRole('ADMIN','BID_LEAD','BID_SENIOR','SALES','BID_SPECIALIST')` | admin, bid_admin, bid_lead, bid_senior, sales, bid_specialist |
| `hasAuthority('retrospective.submit')` | 持该 menuPermission 的角色：bid_lead, bid_admin, bid_senior, bid_specialist（catalog 定义） |

**结论**：字面注解统计会严重失真。下文所有"代码有效"列均按上表计算。
