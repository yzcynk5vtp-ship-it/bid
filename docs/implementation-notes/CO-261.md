# Implementation Notes: CO-261

## Bug Fix: 标讯分配项目负责人后未收到提醒

> Linear: CO-261 | PR: #796 | 合并 commit: `207d182a2` | 日期: 2026-06-18

## 问题描述

销售（sales）反馈：标讯分配项目负责人后收不到任何提醒。具体表现为：
1. 通知面板/铃铛角标对 sales 角色全部 403，打不开。
2. 自动分配负责人、转派负责人成功后，被分配人收不到站内通知。

## Root Cause（两个独立根因）

### 根因一：NotificationController 类级权限锁死

`NotificationController` 类级注解为 `@PreAuthorize("hasRole('ADMIN')")`，导致**所有**读取端点（列出本人通知、查未读数、标记已读等）都被 ADMIN 守卫拦截。

sales 角色经 `RoleProfileCatalog.securityCompatLegacyRole` 映射后 authority = `ROLE_MANAGER`（无 `ROLE_ADMIN`），因此访问通知面板/铃铛全部 403——这是用户"收不到提醒"最直接的体感来源。

### 根因二：分配/转派成功后不调用通知服务

- `TenderCommandService.tryAutoAssign`（CRM 匹配自动分配负责人）成功后只写 `log.info`，不发通知。
- `TenderTransferService.transfer`（管理员转派）成功后同样只写日志。

两个核心路径都缺"通知被分配人"这一步。

## Decision

### 根因一修复：类级注解 `hasRole('ADMIN')` → `isAuthenticated()`

- 类级 `@PreAuthorize` 改为 `isAuthenticated()`：所有已认证用户可读自己的通知（后端按 `currentUser.getId()` 过滤数据，无越权风险）。
- `POST /api/admin/notifications`（管理员推送通知入口）**保留**方法级 `@PreAuthorize("hasRole('ADMIN')")` 守卫。
- **不能直接删除类级注解**：`ArchitectureTest` RULE 15 强制每个 `@RestController` 必须有类级 `@PreAuthorize`。`isAuthenticated()` 既满足 RULE 15，又放开读取端点。

### 根因二修复：新增 `TenderAssignmentNotifier` 外壳

新增 `TenderAssignmentNotifier`（Service 层 `@Component`，59 行）封装两种通知：

- `notifyAutoAssigned(Tender tender)` — 自动分配成功，收件人 = `tender.getProjectManagerId()`，createdBy 兜底为系统(1L)。
- `notifyTransferred(Tender, newOwnerId, oldOwnerName, operatorName, operatorId)` — 转派成功，收件人 = 新负责人。

通知标题使用「【标讯分配】」「【标讯转派】」前缀，便于用户在通知面板快速识别来源。

调用点：
- `TenderCommandService.tryAutoAssign` 的 `log.info(...)` 之后调用 `assignmentNotifier.notifyAutoAssigned(tender)`。
- `TenderTransferService.transfer` 的 `log.info(...)` 之后调用 `assignmentNotifier.notifyTransferred(...)`。

## Tradeoffs

### 1. Notifier 契约：内部 try-catch 吞异常 vs. 调用方包 try-catch

`TenderAssignmentNotifier.send(...)` 内部 `try-catch (RuntimeException)` 吞掉异常并 `log.warn`，**保证通知失败不影响调用方事务**。这与 `BatchTenderAssignAppService` 的"双保险 try-catch"思路一致，但契约更清晰：调用方不需要再包 try-catch。

**取舍点**：吞异常意味着通知失败时调用方无感知（只有日志）。当前权衡认为"分配/转派事务成功"比"通知必达"更重要，通知失败可由运维查日志补救。若未来需要通知必达，应改为事件 + 重试队列，而非让调用方感知异常。

测试上，原 `NotificationFailureShouldNotRollbackAssignment` / `transfer_NotificationFailureShouldNotRollback` 两个"模拟通知异常验证不回滚"的测试因此不再适用（Notifier 契约已保证不抛异常），改由 `TenderAssignmentNotifierTest` 的 `notifyAutoAssigned_NotificationThrows_Swallowed` / `notifyTransferred_NotificationThrows_Swallowed` 专项覆盖异常吞没契约。

### 2. 死代码 `TenderPendingAssignmentNotifier` 未动

仓库存在未被调用的 `TenderPendingAssignmentNotifier`（疑似历史遗留）。按"只改与问题直接相关的代码"原则，本次**未动**（用户决策 4a）。如需清理应单开任务。

### 3. 行预算门禁倒逼抽取

`TenderCommandService` 原本 293 行，加入通知调用后到 323 行（>300 行预算门禁）。解法不是"塞回去"，而是**提取 `TenderAssignmentNotifier`** 让两个 Service 共用——既解决行预算，又消除 `TenderCommandService` / `TenderTransferService` 各自写通知逻辑的重复。最终 `TenderCommandService` 回到 296 行。

### 4. 测试用反射断言 `@PreAuthorize` 值，而非 `@SpringBootTest`

`NotificationControllerPermissionTest` 用反射读取类级/方法级 `@PreAuthorize` 注解的 `value()` 字符串做断言，而不是 `@SpringBootTest` + `MockMvc` 跑真实请求验证 403/200。

**原因**：早期尝试 `@SpringBootTest` + `standaloneSetup` 验证 `@PreAuthorize` 行为失败——`standaloneSetup` 不走 Spring Security filter chain，`@PreAuthorize` 不生效；走完整 `@AutoConfigureMockMvc` 又因 `@AuthenticationPrincipal` 解析复杂导致测试加载失败。反射断言轻量、稳定、直接验证"注解写对了"，配合 `ProjectAccessGuardCoverageTest`（已有集成层覆盖）足够。

## 可复用模式（给后续任务的沉淀）

1. **RULE 15 下放开 Controller 读取权限的正确姿势**：不要删类级 `@PreAuthorize`，改成 `isAuthenticated()`。数据隔离交给 Service 层按 `currentUser.getId()` 过滤。
2. **通知/审计类"副效应"外壳模式**：当多个业务路径都要发同类通知，且要求"失败不影响主事务"时，提取一个 `XxxNotifier` `@Component`，内部 try-catch 吞异常。调用方一行调用，契约清晰。参照 `TenderAssignmentNotifier` / `BatchTenderAssignAppService`。
3. **行预算超限时优先提取协作对象**，不要在原文件硬塞或拆无意义的私有方法。提取出的对象往往本身就有独立测试价值。
4. **验证 `@PreAuthorize` 注解值用反射断言**（轻量稳定），行为层覆盖交给 `ProjectAccessGuardCoverageTest` 这类集成测试。
