# 外部业务系统对接技术设计与接口规范说明

> **西域数智化投标管理平台（以下简称“投标系统”）** 采用**“真实 API 单一路径”**的交付与联调开发模式。为了满足客户多套业务系统（如 OA、CRM、ERP、标讯平台等）的对接需求，本说明提供投标系统现有接口文件获取方式、外部系统向投标系统推送数据的技术方案、字段含义、是否必填及入参示例。

---

## 一、 投标系统现有接口文档获取方式

投标系统已集成 `springdoc-openapi`，自动生成了标准 OpenAPI 3.0 / Swagger 接口规范。外部系统开发人员可以通过以下方式查看和调用我们现有的 **116 个 REST 接口端点**（涵盖用户与权限、标讯匹配、项目组、标书 Agent 编排、资质知识库等）：

### 1. 访问与导入入口

| 用途 | 本地/测试环境 URL | 说明 |
| :--- | :--- | :--- |
| **Swagger UI 可视化门户** | `http://127.0.0.1:18080/swagger-ui.html` | 供外部技术人员在线浏览、检索与单接口调试。 |
| **OpenAPI JSON 规范文件** | `http://127.0.0.1:18080/v3/api-docs` | 包含完整接口结构的机器可读配置，可直接导入 **Postman / Apifox** 或用于自动生成 SDK 客户端。 |
| **OpenAPI YAML 规范文件** | `http://127.0.0.1:18080/v3/api-docs.yaml` | 同上，YAML 格式。 |

> *注：生产环境只需将 `127.0.0.1:18080` 替换为部署域名/IP即可，路径保持完全一致。*

### 2. 鉴权与调用约定

- **鉴权方式**：所有受保护的业务接口均采用 **JWT Bearer Token** 鉴权。
- **调用流程**：
  1. 调用登录接口 `POST /api/auth/login` 换取 token。
  2. 在后续请求中，将 Token 放入 HTTP 请求头的 `Authorization` 字段中：`Authorization: Bearer <jwt-token>`。
- **响应体规范**：接口统一使用 `ApiResponse<T>` 格式包装返回：
  ```json
  {
    "success": true,
    "code": "200",
    "message": "操作成功",
    "data": { ... }
  }
  ```

## 二、 业务系统数据推送投标系统：设计方案与接口定义

为了方便客户的各套业务系统（如 OA、CRM、外部标讯平台等）将业务数据或审批流状态推送给我们，我们基于**“高解耦、高幂等、易维护”**的原则，设计了以下三套对接接口规范。

### 方案 1：外部标讯/商机数据推送接口（用于推送新商机、新标讯数据）

当客户的外部商机系统或标讯抓取系统发现新标讯时，可通过该接口主动推送数据给投标系统。

#### 1. 接口定义
- **请求方式**：`POST`
- **请求路径**：`/api/integration/tenders/push`
- **内容类型**：`application/json`

#### 2. 请求字段说明

| 字段名 | 类型 | 必须 | 字段含义说明 | 示例值 / 格式 |
| :--- | :--- | :--- | :--- | :--- |
| **sourceSystem** | String | **是** | 来源系统编码，用于区分是哪套业务系统推送的 | `"CRM"` 或 `"OA"` |
| **sourceId** | String | **是** | 来源系统中的数据唯一 ID（用于保证接口重试时的幂等性） | `"OPP-2026-00918"` |
| **title** | String | **是** | 标讯/项目/招标项目名称 | `"西域集团2026年度MRO物料采购招标项目"` |
| **customerName** | String | **是** | 客户/招标人单位全称 | `"西域智慧供应链（上海）股份公司"` |
| **publishDate** | String | **是** | 标讯发布时间（格式：YYYY-MM-DD） | `"2026-05-26"` |
| **dueDate** | String | **是** | 投标截止时间（格式：YYYY-MM-DD HH:mm:ss） | `"2026-06-15 10:00:00"` |
| **budgetAmount** | BigDecimal | 否 | 项目预算金额（单位：元） | `5000000.00` |
| **contactPerson** | String | 否 | 客户联系人姓名 | `"张经理"` |
| **contactPhone** | String | 否 | 客户联系人电话 | `"13800138000"` |
| **contentDesc** | String | 否 | 招标需求描述或公告正文 | `"采购范围包含密封件、五金工具及防爆灯具一批..."` |
| **attachments** | Array | 否 | 附件列表（如有招标文件，可一并推送） | 见明细结构 |
| ↳ *fileName* | String | 否 | 附件名称 | `"招标公告及要求.pdf"` |
| ↳ *fileUrl* | String | 否 | 附件下载的公网或局域网免签 URL | `"https://oss.ehsy.com/files/req.pdf"` |

#### 3. 入参示例 (JSON)
```json
{
  "sourceSystem": "CRM",
  "sourceId": "OPP-2026-00918",
  "title": "西域集团2026年度MRO物料采购招标项目",
  "customerName": "西域智慧供应链（上海）股份公司",
  "publishDate": "2026-05-26",
  "dueDate": "2026-06-15 10:00:00",
  "budgetAmount": 5000000.00,
  "contactPerson": "张经理",
  "contactPhone": "13800138000",
  "contentDesc": "采购范围包含密封件、五金工具及防爆灯具一批，具体规格详见招标文件。",
  "attachments": [
    {
      "fileName": "招标公告及要求.pdf",
      "fileUrl": "https://oss.ehsy.com/files/req.pdf"
    }
  ]
}
```

#### 4. 响应体示例
```json
{
  "success": true,
  "code": "200",
  "message": "标讯推送接收成功，系统已建立同步关联",
  "data": {
    "tenderId": "TEN-99887766",
    "status": "RECEIVED"
  }
}
```

### 方案 2：外部流程审批状态回调/推送接口（如 OA 盖章/报销流程结束后回调）

当投标系统把审批（例如标书盖章申请、保证金借款申请）提交到客户 OA系统后，OA 流程结束时，需要将审批结果推送回投标系统。

#### 1. 接口定义
- **请求方式**：`POST`
- **请求路径**：`/api/integration/workflows/callback`
- **内容类型**：`application/json`

#### 2. 请求字段说明

| 字段名 | 类型 | 必须 | 字段含义说明 | 示例值 / 格式 |
| :--- | :--- | :--- | :--- | :--- |
| **sourceBillNo** | String | **是** | 投标系统发起流程时的业务单据号（对应投标系统的关联ID） | `"BID202604300001"` |
| **workflowId** | String | **是** | 外部流程定义 ID（由西域系统或客户系统约定的映射编码） | `"OA_STAMP_FLOW_01"` |
| **requestId** | String | **是** | 外部流程实例 ID（如 OA 生成的 requestId） | `"9954321"` |
| **status** | String | **是** | 审批最终状态。枚举值范围：`ARCHIVE`（归档/通过）, `REJECT`（驳回）, `TERMINATE`（撤销） | `"ARCHIVE"` |
| **operatorNo** | String | **是** | 最终审批人或归档操作人工号 | `"10000"` |
| **comment** | String | 否 | 审批意见或备注说明 | `"经核对，标书盖章版本无误，同意归档。"` |
| **contractFiles** | Array | 否 | 归档盖章后的合同/标书回签附件列表（流程结束后反填） | 见明细结构 |
| ↳ *fileName* | String | 否 | 文件名称 | `"盖章回签版投标文件.pdf"` |
| ↳ *fileUrl* | String | 否 | 盖章回传文件下载的 URL | `"https://oa.ehsy.com/files/signed_tender.pdf"` |

#### 3. 入参示例 (JSON)
```json
{
  "sourceBillNo": "BID202604300001",
  "workflowId": "OA_STAMP_FLOW_01",
  "requestId": "9954321",
  "status": "ARCHIVE",
  "operatorNo": "10000",
  "comment": "经核对，标书盖章版本无误，同意归档。",
  "contractFiles": [
    {
      "fileName": "盖章回签版投标文件.pdf",
      "fileUrl": "https://oa.ehsy.com/files/signed_tender.pdf"
    }
  ]
}
```

#### 4. 响应体示例
```json
{
  "success": true,
  "code": "200",
  "message": "回调流程状态更新成功"
}
```

### 方案 3：组织架构与主数据同步（推荐采用“事件消息通知 + 接口回查”方案）

对于人员信息（`BaseOssUser`）、部门信息（`BaseOssDept`）的推送同步，为保证数据一致性与幂等性，投标系统推荐使用**“事件订阅 + 回查主数据 API”**的推送模式。

- **核心逻辑**：外部系统在人员/部门变更时，向事件库/Kafka 发送一个轻量级“变更事件”（Payload 仅含主键）。投标系统消费到事件后，**反向调用外部系统提供的组织架构查询 API**，拉取最新、最完整的用户信息进行本地更新。
- **好处**：避免事件消息发生乱序、丢失或内容残缺，确保双方数据处于最新的单一可信源。

#### 1. 简易事件推送结构 (MQ / HTTP 事件源推送)

外部系统可以采用 MQ 推送，或者调用投标系统的事件接收入口 `POST /api/integration/events/push` 推送以下事件：

#### 2. 事件请求字段说明

| 字段名 | 类型 | 必须 | 字段含义说明 | 示例值 / 格式 |
| :--- | :--- | :--- | :--- | :--- |
| **traceId** | String | **是** | 全局调用链追踪 ID（方便排查消息丢失/异常链路） | `"tr-88972983792"` |
| **eventTopic** | String | **是** | 事件主题类型。枚举值：`BaseOssUser`（用户变更）, `BaseOssDept`（部门变更） | `"BaseOssUser"` |
| **eventSource** | String | **是** | 事件来源系统 | `"OSS"` / `"HR"` |
| **time** | Long | **是** | 事件发生的时间戳（毫秒级） | `1779786348000` |
| **key** | String | **是** | 业务主键标识（如员工 ID、部门 ID） | `"USER-10029"` |
| **data** | Object | **是** | 消息载体，仅需要包含业务主键 | `{"userId": "USER-10029"}` |

#### 3. 入参示例 (JSON)
```json
{
  "traceId": "tr-88972983792",
  "eventTopic": "BaseOssUser",
  "eventSource": "OSS",
  "time": 1779786348000,
  "key": "USER-10029",
  "data": {
    "userId": "USER-10029"
  }
}
```

#### 4. 投标系统接收后对账逻辑说明

投标系统获取到 `USER-10029` 标识后，将通过反查外部系统接口获取完整的员工信息（如：员工号、姓名、邮箱、部门、手机、状态等），并在本地执行 `upsert`。如果员工状态为“禁用”或“离职”，投标系统将该用户标记为不可用，**不进行物理删除**以维护业务关联的完整性。

## 三、 对接设计规范与建议（给客户技术团队的建议）

1. **接口幂等设计**：
   - 所有的推送接口均设计有业务主键（如 `sourceId`、`sourceBillNo`、`key`）。
   - 客户系统即使因网络超时重试发送相同的数据，投标系统将基于业务主键做幂等性校验（如果已存在则做更新操作，不重复创建记录），防范系统重复消费、重复插库。
2. **状态机控制**：
   - 业务推送和回调遵循状态只向前流转的原则（如流程状态 `DRAFT` -> `PENDING` -> `ARCHIVE`）。
3. **安全防刷与防篡改**：
   - 生产环境中，系统将使用动态 API Key（服务账号机制）和 HTTPS 加密网络来防范接口仿冒及数据泄露。
