# 外部系统集成经验教训

> 本文档记录与外部系统（CRM、组织架构等）对接过程中遇到的问题和经验教训，避免类似问题再次发生。

## CO-277 / CO-278：CRM 回调格式不一致导致"假成功"

### 问题概述

**发现时间**：2026-06-22
**影响范围**：标讯弃标/投标回调不生效
**根因**：两条回调路径使用了不同的 payload 格式，CRM 只处理其中一种

### 时间线

| 时间 | 事件 | 问题 |
|---|---|---|
| 最初设计 | 系统有两条回调路径：§4.1 `tender.status_changed` 和 §4.2 `project.result_confirmed` | 两条路径用了不同的 payload 格式 |
| 2026-06-21 | 4 个项目通过「项目结果登记」页面重新登记 → 回调成功 | 只验证了 §4.2 路径，没验证 §4.1 路径 |
| 2026-06-22 | 新项目通过「评估审核→弃标」触发 → 回调不工作 | §4.1 路径的 payload 格式 CRM 不处理 |
| CRM 响应 | `{"code":"0","msg":"success","data":null}` | `code:0` 是"收到请求"，`data:null` 是"没处理任何数据" |

### 问题症结

#### 症结 1：接口契约的理解偏差

**我们的假设**：CRM 支持 `tender.status_changed` 格式（因为我们文档里这么写了）

**实际情况**：CRM 只实现了 `bidInfoSync` 接口，期望 `bidInfoList` 格式

**证据**：
```json
// §4.1 发送的 payload（CRM 不处理）
{"event":"tender.status_changed","tenderId":334,"sourceId":"296","oldStatus":"EVALUATED","newStatus":"ABANDONED"...}

// §4.2 发送的 payload（CRM 处理）
{"bidInfoList":[{"code":"CC20260622328","name":"测试新建标讯状态回传","status":6,"statusEditor":"郑蓉蓉"...}]}
```

**根本问题**：接口文档是我们自己写的，没有和 CRM 团队确认他们实际支持什么格式。

#### 症结 2：`code:0` 的语义误解

**我们的理解**：`code:0` = 业务成功

**CRM 的实际语义**：
- `code:0` = 请求收到，HTTP 层面成功
- `data:null` = 没有处理任何数据（业务层面无效果）

**证据**（task #37 的响应）：
```json
{"code":"0","msg":"success","data":null}
```

这个响应应该被解读为：**"我收到了你的请求，但我不知道怎么处理，所以什么都没做"**。

**根本问题**：我们没有区分"请求成功"和"业务成功"。

#### 症结 3：两条路径独立实现，只验证了一条

系统中存在两条完全独立的回调路径：

| 路径 | 触发场景 | 监听器 | payload 格式 | CRM 是否处理 |
|---|---|---|---|---|
| §4.1 | 评估审核→弃标/投标 | `WebhookEventListener` | `tender.status_changed` | **否** |
| §4.2 | 项目结果登记 | `ProjectResultConfirmedWebhookListener` | `bidInfoSync` | **是** |

**验证情况**：
- ✅ 用户通过「项目结果登记」页面操作 → 走 §4.2 → 成功
- ❌ 没有验证「评估审核→弃标」这条路径 → §4.1 从来没成功过

**根本问题**：我们以为验证了一个场景就等于验证了所有场景。

#### 症结 4：端到端验证缺失

**我们的验证方式**：
1. 看日志：`Webhook sent: statusCode=200`
2. 看响应：`{"code":"0","msg":"success"}`
3. 结论：回调成功

**缺失的验证**：
- 没有去 CRM 端查询商机状态是否真的变化
- 没有对比发送前后的状态差异

**根本问题**：我们验证的是"请求发出去了"，而不是"业务生效了"。

### 修复方案

统一两条路径的 payload 格式，都使用 `bidInfoSync` 格式：

1. `WebhookEventListener` 改为发送 `BidInfoSyncDTO` 格式
2. 复用 `resolveCrmOpportunityCode()` 函数处理 CC 前缀转换
3. 删除无用的 `TenderStatusChangedPayload` 和 `RecommendationPayload`

### 经验教训

#### 教训 1：外部系统的"成功"响应不可信

**原则**：外部系统返回 `code:0` / `200 OK` 只代表"收到请求"，不代表"业务成功"。

**改进措施**：
1. **建立端到端验证机制**：发送回调后，主动查询外部系统状态，确认变化
2. **对 `data:null` 保持警惕**：如果响应中 `data` 为空，应该记录警告日志，而不是标记成功
3. **区分日志级别**：
   - `statusCode=200` → INFO（请求成功）
   - `data=null` → WARN（业务无效果）
   - 商机状态变化 → INFO（业务成功）

#### 教训 2：多路径实现必须逐条验证

**原则**：系统中存在 N 条业务路径时，必须验证 N 条，不能只验证 1 条就下结论。

**改进措施**：
1. **在设计文档中列出所有业务路径**
2. **每条路径编写独立的测试用例**
3. **验收时逐条勾选**，不能遗漏

#### 教训 3：接口契约必须双方确认

**原则**：接口文档不能只由一方编写，必须双方确认并验证。

**改进措施**：
1. **对接前**：与外部系统团队确认他们实际支持的接口格式
2. **对接中**：通过实际调用验证，而不是只看文档
3. **对接后**：建立契约测试，确保双方实现一致

#### 教训 4：日志要能区分"请求成功"和"业务成功"

**当前日志**：
```
Webhook sent: taskId=37, statusCode=200, responseBody={"code":"0","msg":"success","data":null}
```

**改进后日志**：
```
Webhook request sent: taskId=37, statusCode=200
Webhook response parsed: code=0, msg=success, data=null
WARN: CRM returned data=null, business effect uncertain
```

## 检查清单

### 外部系统对接检查清单

每次对接外部系统时，必须确认：

- [ ] **接口格式确认**：通过实际调用验证外部系统支持的格式（不能只看文档）
- [ ] **响应语义确认**：`code:0` 代表什么？`data:null` 代表什么？
- [ ] **端到端验证方法**：如何确认业务真的生效了？（查询外部系统状态？）
- [ ] **错误处理机制**：失败时如何重试？如何回滚？
- [ ] **监控告警**：如何检测"假成功"的情况？

### 多路径验证检查清单

系统中存在多条业务路径时，必须：

- [ ] **列出所有路径**：在设计文档中明确列出
- [ ] **逐条编写测试**：每条路径有独立的测试用例
- [ ] **逐条端到端验证**：每条路径都验证实际效果
- [ ] **记录路径差异**：不同路径的触发条件、payload 格式、处理逻辑

### 代码审查关注点

审查涉及外部系统对接的代码时，关注：

- [ ] 是否区分了"请求成功"和"业务成功"？
- [ ] 对 `data:null` 的响应是否有警告日志？
- [ ] 是否有端到端验证机制？
- [ ] 是否所有业务路径都已验证？
