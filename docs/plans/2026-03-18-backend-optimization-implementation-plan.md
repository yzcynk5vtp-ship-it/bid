# 后端优化实施方案（2026-03-18）

## 背景

本方案基于当前 `xiyu-bid-poc` 后端代码审查结果制定，目标是优先消除已识别的 P0/P1 风险，并以最小业务扰动的方式提升后端性能、稳定性、安全性与后续可维护性。

本次执行遵循 `Everything Claude Code 标准作业流程`：

1. 规划：形成风险矩阵、实施计划、验收标准与任务清单
2. 开发：优先实现 P0 改造，并用测试约束行为
3. 质检：做代码审查与验证
4. 维护：在不扩散范围的前提下做必要清理

## 风险矩阵最终版

| 编号 | 问题 | 触发条件 | 当前影响 | 风险等级 | 本轮动作 |
| --- | --- | --- | --- | --- | --- |
| 1 | `DashboardAnalyticsService` 性能问题 | 数据量增长到 1k+，分析接口频繁访问 | 全表扫描、JVM 内存聚合、N+1 查询，响应时间与数据库压力同步恶化 | HIGH - P0 | 本轮优先改造 |
| 2 | AI 异步执行方式不一致 | 并发 AI 请求、线程池耗尽、异常传播 | `@Async` 与 `CompletableFuture.runAsync()` 双重异步导致线程池、事务与异常边界不清晰 | HIGH - P0 | 本轮优先改造 |
| 3 | 缓存配置不一致 | 多实例部署、缓存穿透、失效不一致 | 配置声明 Redis，但 analytics 模块实际使用本地 `SimpleCacheManager` | MEDIUM/HIGH - P1 | 本轮方案落地，下阶段实现 |
| 4 | 生产安全配置过宽 | 暴露到公网或半公网环境 | `prometheus` 与错误信息回显策略偏宽，存在信息暴露面 | MEDIUM - P1 | 本轮方案落地，下阶段实现 |
| 5 | 定时告警服务含 placeholder/TODO 逻辑 | 定时任务持续执行 | 误报、脏告警、全表扫描与不可预测的数据副作用 | MEDIUM - P1 | 下阶段实现 |
| 6 | 架构门禁仅覆盖新模块 | 新功能继续开发 | 新老分层共存，公共根包继续膨胀，长期维护成本上升 | LOW/MEDIUM - P2 | 持续治理 |

## 分阶段实施计划

### Phase 0：本轮立即执行

目标：优先收口最容易演变为线上事故的 P0 项。

范围：

1. `DashboardAnalyticsService` 性能重构
2. AI 异步执行统一化

执行策略：

1. 先保行为不变，再调整实现
2. 优先覆盖 `overview` 和关键 drill-down 路径
3. 用测试锁定输出结构与关键聚合结果
4. 所有改动尽量限制在 analytics、ai、async 配置相关文件

### Phase 1：一周内完成

目标：完成本轮代码合并、回归验证与上线准备。

范围：

1. 完成 P0 改造代码合并
2. 补充或更新测试
3. 增加必要的运行说明与风险备注
4. 产出性能基线和验证结果

### Phase 2：一个月内完成

目标：收口核心 P1 风险，降低部署与运行期不确定性。

范围：

1. 缓存统一为 `RedisCacheManager`
2. 收紧生产安全配置
3. 重构定时告警服务为真实查询链路
4. 补索引、慢查询基线、关键接口压测

### Phase 3：持续治理

目标：防止单体继续失控扩张。

范围：

1. 升级 ArchUnit 门禁，让所有新增代码必须走模块包
2. 冻结根包 `controller/service/repository/entity` 的继续扩张
3. 按模块逐步把老代码迁入边界清晰的子包

## 每项改造的验收标准

### 1. Dashboard/Analytics 性能改造

验收标准：

1. `DashboardAnalyticsService` 不再依赖多个核心路径上的 `findAll()` 全表拉取作为主实现
2. 消除明显的循环内 `findById()` / 按项目逐个查询附件、导出的 N+1 模式
3. 关键接口输出结构与字段语义保持兼容
4. 新增或更新测试，覆盖 overview 或核心 drill-down 的行为正确性
5. 代码中能清晰看出“聚合查询/批量查询优先”的实现方向

完成定义：

1. 关键方法具备 repository 级聚合或批量数据装载
2. 至少一组 analytics 测试通过
3. 不引入跨模块耦合扩散

### 2. AI 异步统一化

验收标准：

1. `AiService` 不再出现 `@Async` + `CompletableFuture.runAsync()` 的双重异步
2. AI 异步调用统一走 Spring 管理的执行器
3. 异常传播、日志记录与返回类型保持清晰一致
4. 保留现有对外 API 语义，避免影响调用方
5. 补充或更新测试，验证异步方法与同步方法行为一致

完成定义：

1. 仅保留一层异步调度模型
2. 线程池使用路径明确
3. 相关测试通过

### 3. 缓存统一化

验收标准：

1. analytics 模块不再自定义与全局配置冲突的本地 `CacheManager`
2. dashboard 缓存统一使用 Redis TTL 与 key prefix 配置
3. 具备显式失效策略，避免脏缓存长期存在

### 4. 生产安全配置收紧

验收标准：

1. `management` 暴露面最小化，Prometheus 仅允许受控访问
2. 生产环境错误信息回显策略不再暴露过多内部细节
3. 变更后不影响现有健康检查与运维采集

### 5. 定时告警服务重构

验收标准：

1. 移除 placeholder/TODO 作为核心判断逻辑
2. 定时任务不再依赖全表扫描作为默认实现
3. 告警生成具备幂等或去重策略
4. 能区分真实异常、误报和数据缺失

### 6. 架构门禁升级

验收标准：

1. 新增代码默认必须落入模块子包
2. ArchUnit 对老模块逐步扩大覆盖面
3. 根包公共层不再成为新功能默认落点

## 可直接分配给开发的任务清单

### Task A：Analytics 性能改造

负责人：后端开发 A

任务内容：

1. 梳理 `DashboardAnalyticsService` 中所有 `findAll()`、循环查库与 N+1 热点
2. 优先重构 `getOverview()` 及其直接依赖方法
3. 评估是否新增 analytics 专属 repository 查询接口
4. 保持 DTO 返回结构不变
5. 补 analytics 单测或集成测试

交付物：

1. 改造后的 service/repository 代码
2. 对应测试
3. 变更说明：哪些查询被批量化/聚合化

3. 清理重复或模糊的 async 配置
4. 补 AI service 测试，验证异常与返回语义

交付物：

1. 改造后的 `AiService` 与 async 配置
2. 对应测试
3. 变更说明：线程池与事务边界如何统一

### Task C：缓存统一设计

负责人：后端开发 C

1. 盘点当前缓存实现与全局配置冲突点
2. 设计 Redis 统一缓存方案
3. 输出 cache key、TTL、失效策略清单

交付物：

1. 技术设计说明
2. 待开发任务拆分

### Task D：生产安全配置收紧

负责人：后端开发 D / 运维协作

任务内容：

1. 收紧 actuator 暴露面
2. 调整生产环境错误返回策略
3. 校验监控与探针不受影响

交付物：

1. 配置变更方案
2. 回归检查清单

### Task E：定时告警服务重构

负责人：后端开发 E

## 本轮执行范围

本轮开发只落地 Task A 与 Task B。

原因：

1. 两项均为 P0，且能直接降低近期线上风险
2. 两项文件边界清晰，适合并行开发
3. 其余任务更适合在本轮完成后按方案继续推进
