# 下一阶段整改路线图（2026-03-18）

## 背景

根据验收报告与当前工作区代码复核结果，项目的后续整改需要按“发布可信度优先、权限治理深化、自动化稳定性收口”三条主线推进。需要特别说明的是，当前源码与验收报告存在证据链偏差：

- 认证链路仍以浏览器 `localStorage/sessionStorage` 保存 token 为主
- 后端 `refresh/logout` 尚未具备服务端会话失效语义
- `system_settings` 的 Flyway `V13` 仅存在于编译产物，不在源码迁移目录
- 前端系统设置页仍以本地 `ref` 数据为主，未形成完整可追溯的真实 API 闭环

因此，本路线图优先以“恢复源码可发布性与发布链路可信度”为原则。

## P0：发布可信度与安全基线收口

### 目标

先把会影响上线可信度和发布一致性的项收口，确保源码、迁移、认证语义至少形成可审计的主链路。

### 可执行清单

1. 补齐源码内正式 `system_settings` Flyway 迁移
   - 把编译产物中的 `V13__create_system_settings_table.sql` 回补到 `backend/src/main/resources/db/migration-mysql/`
   - 补充 Flyway 验证，确保空库启动时 baseline 之后继续执行 `V13`
   - 复核 MySQL baseline 与 `V13` 的执行顺序和不冲突性

2. 设计并落地服务端 refresh session 基础设施
   - 新增 refresh session 持久化表、实体和 repository
   - 登录时签发 access token + 生成 refresh session
   - 刷新时校验并轮换 refresh session
   - 登出时撤销当前会话，而不再只是记录日志

3. 前端认证承载改为“内存 access token + 服务端 refresh”
   - 移除运行主链路对 `localStorage/sessionStorage token` 的依赖
   - 调整 axios 401 处理为 refresh-and-retry
   - 让路由守卫依赖真实 session restore，而不是存储态兜底

### 预计产出

- 源码可审计的 `V13` migration 与 Flyway 验证
- 服务端 refresh/revoke 会话主链路
- 前端认证主链路从浏览器持久化 token 升级为更稳妥的会话方案
- 一组可复用的认证与迁移验证测试

## P1：权限治理从页面控制推进到数据强约束

### 目标

把当前“菜单/路由可见性控制”升级为“后端接口与查询层的真实数据范围控制”。

### 可执行清单

1. 建立统一的数据权限模型
   - 明确 `dataScope`、`allowedProjects`、`allowedDepts` 的服务端语义
   - 梳理角色、部门、项目组之间的授权继承规则

2. 把数据范围约束下沉到后端接口
   - 在项目、分析、审计等关键列表接口增加当前用户上下文过滤
   - 对越权查询与越权变更增加服务端拒绝逻辑

3. 打通系统设置页到真实权限治理接口
   - 补系统配置、角色权限、部门权限、流程映射的后端接口与前端调用
   - 消除当前设置页对页面本地状态的强依赖

### 预计产出

- 可追溯的权限配置模型
- 关键业务接口的数据级权限控制
- 系统设置页与真实后端配置的闭环
- 权限矩阵专项测试

## P2：自动化稳定性与测试资产统一

### 目标

把现有“能跑”的 E2E 体系收敛成稳定、可维护、低串扰的回归资产。

### 可执行清单

1. 处理 `auth-access-control` 并发偶发失败
   - 排查登录态注入、缓存、全局状态和测试用户隔离问题
   - 收敛 flaky case 的触发条件与复现方式

2. 统一认证和测试数据辅助
   - 抽统一登录 helper
   - 统一测试数据创建、清理和命名规则
   - 清理各 spec 内重复的 session 注入代码

3. 标准化运行时隔离策略
   - 统一前端缓存、浏览器 storage、后端测试数据的隔离方式
   - 明确串行/并发执行边界

### 预计产出

- 更稳定的 Playwright 全量回归
- 可复用的登录与数据构造辅助层
- 降低新增用例维护成本

## 本轮开发任务

本轮按标准作业流程优先执行 P0 的最小闭环：

1. 规划：确认“源码缺失的 `V13` migration 回补 + Flyway 验证补强”为当前最小可交付切口
2. 开发：先补验证，再补迁移源码
3. 质检：检查迁移命名、执行顺序、空库兼容性
4. 维护：保持变更最小，不顺带扩散到未证实的系统设置应用层代码
