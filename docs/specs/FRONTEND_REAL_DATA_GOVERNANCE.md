# Frontend 真实数据交付红线协议

一旦我所属的文件夹有所变化，请更新我。

本文件取代了原有的《双模式妥协期数据源治理约定》，确立了距离上线交付冲刺期最后 2 个月前端开发的刚性红线：
**系统已全量舍弃本地 Mock Demo 体系，坚决转向 100% 真实后端数据双向通信。**

## 1. 唯一合法法则

在目前的 `API 模式` (也是项目当前唯一存活模式) 之下，前端应用：
- 只能渲染来自后端的真实持久化数据。
- 遇后端空态，必须展示真实的 `EmptyState` 空状态骨架。
- 遇后端接口延迟报错，必须抛给用户真实的失败交互提示框。
- 遇尚未接入（即后端尚未给出 Swagger 契约）的占位节点，统一硬编码前端防触碰屏蔽，并打上 `Coming Soon` 标。

**绝不允许偷偷给组件塞假数据粉饰太平。**

## 2. 绝对禁区

以下所有涉及过往 POC 时期快速演示的技术债行为，一经门禁探测直接认定为构建违规：
- **禁止** 任何文件导入甚至隐式引用 `@/api/mock` 或 `mockData` 静态数据字典。
- **禁止** 组件试图读写 `demoPersistence` 进行前端自导自演。
- **禁止** 在 `store` 层进行“如果 API 没拉到数据，我就先塞两条演示数组”的恶劣兜底掩盖行为。
- **禁止** API 模块拦截非真实后端的返回值并伪造 200 HTTP 报文。

## 3. CI/CD 红线

前置构建防腐脚本 `npm run check:front-data-boundaries` 的目标已切换：从原本的“限制 mock 边界”变为了此刻的“彻底清剿遗留 mock 污染”。

- 打回一切 `// TODO: switch to real api later` 相关的假数据。
- 打回任何 `isMockMode()` 及其衍生逻辑判定分支（目前业务只存在唯一的真实分支）。 
- 组件即使崩盘报错，也应当暴露出明确的真实的异常状态给 QA 发现，而不是静默吃掉后显示 Mock 假象迷惑测试人员。
