## 2. 项目目标与实施原则

### 2.1 项目建设目标

围绕西域投标业务数字化转型目标，本项目按2026年7月10日上线目标倒排完成以下目标：

1. 建立投标业务全生命周期线上闭环，覆盖标讯、项目、任务、文档、费用、结果、知识沉淀等主链路。
2. 建立统一的资质库、案例库、模板库、平台账户与费用台账，实现资源可复用、过程可追溯。
3. 建立可视化经营分析体系，为管理层提供中标率、投入产出、区域分布、竞对分析等决策支持。
4. 完成与甲方现有系统的必要集成准备与分阶段联调，支撑后续稳定运营与扩展。
5. 在满足甲方私有化部署、安全审计、权限控制、培训交接要求的前提下，完成正式上线和试运行启动。

### 2.2 实施原则

1. **范围冻结原则**：以正式商用品范围为交付基线，非正式范围或演示能力不进入本次上线清单。
2. **蓝图先行原则**：先完成业务流程、权限口径、集成边界和数据口径确认，再进入开发与配置。
3. **联调前置原则**：CRM、OA、组织架构、消息提醒等对接项需在实现阶段前完成接口确认。
4. **测试门禁原则**：SIT、UAT、上线演练逐级通过后，方可进入生产切换。
5. **文档闭环原则**：关键设计、测试、培训、上线成果均形成正式文档，并纳入验收依据。

### 2.3 计划成立的前提假设

本计划按“2026年7月10日完成正式上线”倒排编制，周六、周日及法定节假日不计入有效工作日；2026年劳动节、端午节按国务院办公厅节假日安排剔除。计划成立前提如下：

1. 甲方在项目启动后及时完成项目组建、关键用户指派和环境申请。
2. 蓝图阶段确认后的新增需求，按照变更流程单独评估，不直接挤占本期工期。
3. CRM、OA、组织架构、消息等外部系统能够按计划提供接口文档、测试环境和联调窗口。
4. 本次上线范围以正式版白名单能力为准，演示性能力和未闭环能力不纳入正式验收。
5. 甲方私有云环境、网络、安全策略、域名证书和账号权限能够按计划到位。

## 3. 实施范围基线

结合当前系统代码、交付范围冻结文档及需求清单，本次实施按“正式版白名单能力”组织交付，具体如下：

### 3.1 本次交付范围

1. 认证与权限
   - 登录认证、会话管理、角色权限、系统参数配置、审计日志。
2. 标讯与项目主链路
   - 标讯获取/录入、标讯跟进、标讯 AI 核心分析（对齐演示关键点）、项目立项、任务协同、项目详情、结果录入与查询闭环。
3. 知识资产中心
   - 资质库、案例库、模板库核心管理能力及基础检索调用。
4. 资源管理
   - 费用申请/审批/退还跟踪、平台账户管理、BAR证书及借还管理。
5. 协作与文档
   - 文档章节、协作讨论、版本历史、版本比对、回滚查询。
6. 数据分析
   - 管理驾驶舱（年度/季度分析）、多维分析报表（区域/项目/中标率/投入产出/竞对）、数据穿透下钻。
7. 系统集成（预留与分步实施）
   - CRM 客户同步、OA/审批流（用印/付款）流程映射、组织架构同步。
8. 发布与运维基础能力
   - Flyway 数据库迁移、Spring Actuator 健康检查、Prometheus 指标监控、CI/CD 自动化流水线。
8. 合同与交付资产
   - 本地化部署版本交付、永久使用权交付、源代码与二次开发说明文档交付、运维与培训资料交付。

## 6. 分阶段实施计划

### 6.1 第一阶段：项目启动与业务调研（2026-05-07 ~ 2026-05-22）

**工作目标：** 明确目标、范围、组织机制、关键接口、环境准备要求，形成统一项目基线；在蓝图设计启动后继续开展补充调研与场景复核，避免前期调研时间不足。

**主要工作内容：**

1. 召开项目启动会，明确双方组织、汇报机制、会议机制、变更机制。
2. 梳理甲方现状流程，覆盖标讯管理、项目立项、任务协同、文档编制、费用与账户、结果归档、数据分析。
3. 确认本次交付范围、阶段边界、二期候选项及不纳入本次上线项。
4. 收集甲方基础环境资料，包括网络、私有云资源、安全要求、账号申请、域名、SSL证书、备份策略。
5. 收集CRM、OA、组织架构、企业微信/消息等系统接口资料，明确接口人和联调时间。
6. 形成详细项目主计划、调研清单、项目组织成员清单。
7. 2026年5月7日召开启动会，2026年5月9日开展首场正式客户访谈；在蓝图初稿形成期间，同步复核流程、字段、角色、权限、报表和接口假设。

**阶段交付物：**

1. 《项目组织成员清单》
2. 《投标管理系统项目实施主计划》
3. 《启动会PPT》
4. 《调研流程清单》
5. 《项目问题与风险初始清单》

**甲方配合事项：**

1. 指定项目负责人及关键用户。
2. 提供需求任务书确认版、现状流程资料及接口资料。
3. 安排相关部门参与调研访谈。

**主要工作内容：**

1. 形成投标业务流程图、角色职责矩阵、关键控制点及预警策略。
2. 完成需求适配分析，明确标准功能、配置实现、定制开发、接口实现四类清单。
3. 完成权限设计，覆盖组织、角色、数据范围、审批与审计规则。
4. 完成接口设计，明确CRM、OA、组织架构、消息提醒等对接字段、频率、调用方式和异常处理。
5. 结合甲方业务场景形成AI应用专项方案，明确在标书合规检查、风险预警、知识推荐、历史数据分析等场景中的应用逻辑、预期价值与实施路径。
6. 完成关键页面与关键流程原型确认。
7. 完成蓝图评审会并签字确认，冻结蓝图范围。

5. 《投标管理系统原型设计文档》
6. 《投标管理系统项目业务落地解决方案》

8. 《蓝图汇报PPT》
9. 《蓝图确认函》

1. 蓝图文档经双方评审确认。
2. 定制项与接口项完成优先级排序和实现口径确认。
3. 未确认项不得进入开发承诺。

## 10. 部署架构与服务器配置方案

### 10.1 配置依据

结合当前代码实现与现有配置，项目当前应用技术架构可确认如下：

1. 前端：Vue 3 + Vite，构建产物为静态站点；生产环境由 Nginx 或同类 Web 服务统一承载访问。
2. 后端：Spring Boot 3.2 / Java 21 单体服务，提供 REST API、认证、业务处理及健康检查接口。
3. 数据库：MySQL 8.0，配合 Flyway 承担业务数据存储与数据库版本迁移。
4. 缓存：Redis，主要用于缓存、限流及令牌相关辅助能力。
5. 可观测性：已接入 Spring Boot Actuator，并暴露 Prometheus 指标接口；日志收集与告警由部署环境统一建设。

考虑“约200名用户使用”的场景，正式环境按以下基线进行容量规划：

1. 200名注册/使用用户。
2. 峰值并发用户按40~60人估算。
3. 普通页面响应目标不高于2秒。
4. 主要访问集中于工作台、项目、资源、知识库、分析看板等典型办公场景。
5. 附件与文档存储独立于应用节点部署。

### 10.2 部署拓扑

**正式环境采用三层部署：**

1. 访问层：Nginx/负载均衡
2. 应用层：Spring Boot应用服务
3. 数据层：MySQL 8.0 + Redis + 文件存储

### 10.3 服务器资源配置

以下配置结合当前代码架构、200名用户规模和办公类业务并发特征确定，为实施阶段采用的容量规划基线，并非代码仓库中已固化的实际生产环境。

#### 方案A：正式生产配置

**适用场景：** 正式生产环境、要求较高稳定性、需要一定冗余和可扩展能力。

| 组件 | 配置数量 | 配置标准 | 实施说明 |
|------|----------|----------|----------|
| 访问层节点 | 2台 | 2 vCPU / 4 GB / 50 GB SSD | 部署 Nginx 或同类 Web 服务，承载前端静态资源与反向代理 |
| 应用服务节点 | 2台 | 4 vCPU / 8~16 GB / 100 GB SSD | 部署 Spring Boot 服务，采用双节点分担访问压力 |
| MySQL 主库 | 1台 | 8 vCPU / 16 GB / 500 GB SSD | 核心业务数据库，独立部署 |
| MySQL 备库 | 1台 | 8 vCPU / 16 GB / 500 GB SSD | 用于主备复制或热备切换 |
| Redis 节点 | 1台 | 2~4 vCPU / 4 GB / 50 GB SSD | 独立部署，用于缓存和限流辅助 |
| 文件存储 | 1套 | 初始 500 GB，可扩展 | 采用对象存储、NAS或甲方现有文件服务 |
| 监控日志节点 | 1台 | 2 vCPU / 4 GB / 100 GB SSD | 与甲方运维平台整合，承载指标采集与日志归集 |

#### 方案B：试运行/预生产配置

**适用场景：** 测试环境、预生产环境、短期试运行场景，不作为长期正式生产架构。

| 组件 | 配置数量 | 配置标准 | 实施说明 |
|------|----------|----------|----------|
| 访问层节点 | 1台 | 2 vCPU / 4 GB | 单节点，非高可用 |
| 应用服务节点 | 1台 | 4 vCPU / 8 GB | 单节点部署 |
| MySQL 8.0 | 1台 | 4~8 vCPU / 16 GB / 300 GB SSD | 单库部署，执行强化备份策略 |
| Redis | 1台 | 2 vCPU / 4 GB | 独立部署或由甲方统一提供 |

#### 方案选型原则

1. 本项目上线后承载正式业务时，采用方案A。
2. 当前阶段以联调、预演和短期试运行为主时，采用方案B；正式投产前升级至具备主备和双节点能力的配置。
3. 后续用户规模、导出任务量或文档附件规模明显增长时，优先扩容应用节点、数据库磁盘和文件存储容量。

### 10.4 数据库实施要求

为保障正式环境稳定运行，数据库实施按以下要求执行：

1. **数据库版本与架构**：正式环境统一采用 MySQL 8.0 或以上版本，与当前系统实现保持一致；生产环境采用主库 + 备库模式部署，支撑故障切换和维护窗口切换。
2. **部署原则**：数据库独立部署，不与应用服务、缓存服务混部；数据目录、日志目录、备份目录分区管理。
3. **连接资源控制**：单应用节点数据库连接池控制在 20~30；双应用节点总连接数控制在数据库最大连接数的 70% 以内，避免高峰期连接耗尽。
4. **备份策略**：执行“每日全量备份 + 日志连续归档”策略；全量备份每天至少 1 次，binlog 按 15 分钟级别归档。
5. **恢复指标**：正式环境恢复目标控制为 RPO 不高于 15 分钟、RTO 不高于 2 小时，并在上线演练阶段完成恢复验证。
6. **性能治理要求**：对核心业务表建立必要索引，定期执行统计信息更新、慢 SQL 分析、容量巡检和历史数据归档，确保查询性能与数据增长可控。
7. **变更控制要求**：数据库结构变更统一通过 Flyway 迁移脚本实施，不允许直接在生产环境手工改表。

### 10.5 缓存与中间件实施要求

Redis 作为正式环境辅助中间件，按以下原则部署和管理：

1. **用途定位**：主要承载缓存、限流及令牌相关辅助能力，不作为唯一业务数据存储介质。
2. **部署要求**：独立部署，不与 MySQL 混部；如甲方已有统一缓存平台，则接入甲方现有 Redis 服务。
3. **运行策略**：启用持久化和内存淘汰策略，防止节点重启或高峰期内存不足导致服务抖动。
4. **容量预留**：预留 20%~30% 内存冗余，用于应对业务波峰、缓存重建和短时突发请求。
5. **故障原则**：缓存或限流中间件异常时，应用侧应具备降级或容错能力，不因单一缓存故障导致核心业务整体不可用。

### 10.6 安全、网络与可观测性实施要求

正式环境部署、安全和运行监控应满足以下要求：

1. **访问安全**：系统统一采用 HTTPS 访问，证书由甲方或双方约定方式统一管理。
2. **网络分区**：前端访问层、应用层、数据库层分区部署，数据库不对公网开放，仅允许受控内网访问。
3. **权限与审计**：正式环境启用操作日志审计、访问日志、异常日志和安全日志，满足后续追溯、排障和审计要求。
4. **接口安全**：开放 API 通过网关、反向代理或安全边界设备实施访问控制，至少具备鉴权、频率限制、来源控制和异常拦截能力。
5. **安全合规**：部署方案需满足甲方既有的等保、堡垒机、网络白名单、密码策略、主机加固等规范要求。
6. **运行监控**：应用侧基于 Spring Boot Actuator 和 Prometheus 指标接口输出运行状态，部署环境负责完成日志归集、指标采集、阈值告警和事件通知配置。
7. **上线验证要求**：正式切换后须完成健康检查、关键接口检查、核心业务场景检查和告警链路检查。

### 10.7 非功能性实施指标

以下指标作为正式上线和验收阶段的非功能性基线：

1. **可用性指标**：系统支持 7×24 小时运行，年平均可用性目标不低于 99.5%。
2. **性能指标**：普通操作页面响应时间控制在 2 秒以内，批量导出、复杂统计等长耗时操作采用异步或任务化处理机制。
3. **兼容性指标**：正式环境支持 Google Chrome、Microsoft Edge、Mozilla Firefox 等主流浏览器访问，不依赖浏览器插件或本地客户端。
4. **移动支持指标**：支持移动端关键场景访问，至少覆盖审批、进度查看、预警接收等轻量业务场景，采用企业微信、小程序或移动浏览器适配方式。
5. **集成扩展指标**：系统提供标准 API 接口，并预留与财务系统、电子签章、第三方商机/标讯服务等系统的扩展能力。

1. 培训对象需覆盖管理层、关键用户、最终用户、系统管理员及运维人员。
2. 培训内容除功能操作外，还应包含流程宣贯、权限说明、运维机制及必要的二次开发说明。
3. 若甲方相关人员在培训后未充分掌握系统使用或基础管理能力，乙方应根据需求安排补充培训。
4. 上线后试运行保障期不少于3个月，试运行阶段形成运行跟踪报告和问题跟进台账。

## 16. 结论

基于当前系统基础与需求目标，本项目在“正式范围冻结、蓝图确认前置、接口配合及时、环境准备同步推进”的前提下，采用2026年7月10日上线倒排计划具备可执行性。本项目以2026年4月27日启动准备、2026年5月7日启动会、2026年5月9日首场正式客户访谈、2026年7月10日完成上线为当前基线计划，剔除周末和法定节假日后约53个有效工作日，严格按照里程碑推进，并以UAT通过和上线演练通过作为最终上线门禁。

为确保项目计划书能够直接作为客户沟通与项目启动依据，在正式提交前同步补充以下信息并固化至封面或附录：

1. 双方项目组织成员名单
2. 甲方指定私有云环境清单
3. 集成系统清单及接口责任人
4. 验收签字流程和审批路径
