## 实现说明

### 合同借阅申请

- 新增 `contractborrow` bounded context，按 `domain / application / infrastructure / controller` 分层。
- 纯核心 `ContractBorrowLifecyclePolicy` 只处理状态迁移和规则判定，不访问 DB、不做 DTO 转换。
- 应用服务拆成命令与查询编排，controller 只接收 HTTP 参数并返回统一响应。
- 支持提交、审批、驳回、归还、取消、列表、详情、概览和事件历史。
- 前端新增独立合同借阅页，不再停留在弹窗假入口；工作台快捷入口进入 `/resource/contract-borrow`。

### 标讯真实字段与检索

- `tenders` 表补齐地区、行业、采购方、发布日期、联系人、描述、标签等真实字段。
- `GET /api/tenders` 保持既有 `ApiResponse<List<TenderDTO>>` 返回形态，同时接收服务端筛选参数。
- 原 `TenderService` 保留为兼容门面，真实职责拆到 query/command/mapper/specification。
- 前端 `tendersApi.getList(params)` 透传 query 参数，列表页移除地区/行业等浏览器本地假筛选。
- 手工录入标讯调用 `tendersApi.create()`，由后端真实入库。

### 客户类型分析

- `Project`、`ProjectDTO`、`ProjectRequest`、mapper 支持 `customerType`。
- 历史项目无客户类型时在聚合中归为 `未分类`，不借用行业字段。
- 新增客户类型维度概览和下钻 API，查询、计算、装配拆成独立服务。
- 前端看板通过独立 `CustomerTypePanel.vue` 接入，不继续膨胀 `Dashboard.vue` 的业务逻辑。

## 架构与拆分证明

- FP-Java Profile：核心规则、应用编排、数据访问、DTO 转换、状态写入均拆在不同职责类中。
- Split-First Rule：新增和主要受影响 service 文件均控制在 300 行以内。
- `DashboardAnalyticsService` 被收敛为 183 行门面；指标下钻拆入 274 行的 `DashboardAnalyticsMetricDrillDownService`，客户类型维度使用独立 analytics 服务和组件。
