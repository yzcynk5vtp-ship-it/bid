1. 背景与根因

用户反馈：任务详情只读（view）模式下，附件链接点击无反应。HTML 片段显示 <a href="javascript:void(0)"> 外层 el-upload 带有 aria-disabled="true"。

根因： Element Plus 的 el-upload 在 :disabled="true" 时，会把整个组件（含 #file 插槽内的 <a> 链接）置为 aria-disabled，点击事件被组件吞掉，@click.prevent 不触发，下载链路断裂。

关键教训： 单测里用的是 ElUpload 的 stub，stub 没有实现 disabled 锁逻辑，所以这个 bug 在单测里完全不可见。真实 EP 验证（用真 Element Plus 挂载）才暴露问题。已沉淀到 docs/lessons/lessons-learned.md。

2. 修复方案

沿用已合入 main 的交付物修复（commit e9e073688）的同一思路：把已保存的附件从 el-upload 的 file-list 里移出，单独渲染为独立的 <a class="attachment-link">，与 .deliverable-link 风格一致。

模板改动

el-upload 上加 v-if="!readonly"，移除 :disabled="readonly"
只读模式下新增 .attachment-list 容器，遍历 savedAttachments 渲染 .attachment-link

脚本改动（内联，与 main 结构一致）

拆分原 attachmentFileList 为两个 computed：
- savedAttachments: 已保存（含 id）附件，构造 /api/projects/{pid}/documents/{id}/download URL
- attachmentFileList: 仅保留待上传的 raw File，避免禁用态把已保存附件一起锁死

样式改动

新增 .attachment-list / .attachment-link，复用 .deliverable-link 的视觉

3. 规范外的决策与权衡

决策 A：手工重写而非 cherry-pick（命中豁免条款）

本修复最初在已合入 main 的旧分支 agent/zcode/fix-task-deliverable-dup 上完成（commit cea1ddebc），当时采用了提取 composable useTaskAttachmentList.js 的方案。但 main 随后合入了 0c2ee52f2 refactor(frontend): inline 12 single-use composables，把 useTaskDeliverableDownload 等 12 个 composable 内联回了 TaskForm.vue，与我提取 composable 的方向相反。

cherry-pick cea1ddebc 时在 TaskForm.vue 产生内容冲突（import 区与脚本区）。按 RELIABILITY.md §回退恢复纪律，命中「三种允许手工重写的情况」之第二条——原 commit 的改动语义已被重组（composable 被内联，cherry-pick 后代码位置无意义），故放弃 cherry-pick，手工把修复逻辑应用到 main 的新内联结构上。原 commit hash cea1ddebc 已在此记录，便于未来追溯。

决策 B：只读模式不再 emit attachment-preview 事件

原实现：el-upload @preview → emit('attachment-preview', file) → ProjectTaskBoardCard.handleAttachmentPreview → window.open(url)。

新实现：只读模式 el-upload 不渲染，@preview 不会触发；附件下载改为直接走 handleDownloadAttachment（main 内联的下载函数，axios 带认证下载），与交付物下载保持一致。

收益： 会话过期时不会跳空白页（window.open 拿不到 cookie 会显示 401 空白），更健壮。这是与交付物下载一致的、有意的设计选择。
契约影响： attachment-preview 事件在只读模式不再触发。defineEmits 仍保留该事件（edit 模式 el-upload 仍渲染并挂 @preview），未删 emit 声明，避免破坏 ProjectTaskBoardCard.spec.js 的契约测试（该测试仍通过）。edit 模式附件下载仍走 handleDownloadAttachment（#file 插槽里的 <a> 直接绑 click）。

决策 C：无 id 附件在只读模式不渲染

savedAttachments 过滤 file?.id != null，所以只读模式无 id 的附件不会显示。原行为是渲染但点击提示"文件信息缺失"。新行为是不渲染。这是合理的：只读模式打开的是已落库任务，附件理论上都有 id；无 id 附件属于编辑态未保存的，本就不该在只读模式展示。

5. 待办与风险

  • edit 模式已保存附件下载：edit 模式下已保存附件（含 id）不在 attachmentFileList 里，用户在 edit 模式看不到已保存附件的下载链接。这是潜在的功能缺口，但不在本次 bug 修复 scope 内（用户报告的是只读模式）。建议后续补一个 edit 模式已保存附件渲染条目。
  • attachment-preview 事件清理：如确认只读模式不再需要该事件，可在后续 PR 清理 defineEmits 和 ProjectTaskBoardCard.handleAttachmentPreview。本次保留以最小化改动面。
