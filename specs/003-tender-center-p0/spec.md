## User Scenarios & Testing *(mandatory)*

### User Story 1 - 三段式评估表填写并提交决策 (Priority: P1)

项目负责人在跟踪中标讯进入评估阶段时，需要按 PRD 4.2.5 要求填写包含「基础信息」「客户信息」「投标负责人建议」三段的评估表。目前评估表是单段 7 字段，无法承载客户决策链 13 个角色 × 14 维度信息，导致投标决策依据缺失。

**Why this priority**: 这是 PRD §4.2.5 的核心要求，评估表是「跟踪中 → 已评估 → 投标中/已放弃」状态流转的唯一触发点。三段都齐备才能让投标管理员/组长基于完整客户信息做出投标决策。当前的单段 7 字段评估表完全无法承载 PRD 业务模型，是阻塞合规交付的最高优先级问题。

**Independent Test**: 以项目负责人身份登录 → 进入跟踪中标讯详情页 → 切换到「项目评估表」Tab → 看到 3 段（基础信息 8 字段 + 客户信息 13×14 表格 + 投标负责人建议） → 填写并提交 → 状态变为「已评估」 → 投标管理员收到待办通知 → 查看完整评估并执行立即投标或放弃投标 → 状态正确流转。整个闭环可独立验收。

**Acceptance Scenarios**:

1. **Given** 项目负责人持有「跟踪中」标讯，**When** 打开评估表 Tab，**Then** 系统展示 3 段：基础信息（8 个字段如入围家数、年度采购金额、风险预判等）、客户信息（13 个角色行 × 14 个信息列的固定表格）、投标负责人建议（是否投标 + 理由）
2. **Given** 评估表三段任意一段必填项未填，**When** 点击提交，**Then** 系统拒绝并高亮缺失项
3. **Given** 三段填写完整，**When** 提交，**Then** 状态从「跟踪中」变为「已评估」，投标管理员、组长、投标负责人收到待办通知
4. **Given** 标讯已是「已评估」状态，**When** 项目负责人重新编辑评估表并保存，**Then** 评估表内容更新，状态变为「已评估-待审核」（`EVALUATED_PENDING_REVIEW`），投标管理员/组长收到「需重新审核」待办通知
5. **Given** 标讯处于「已评估-待审核」状态，**When** 投标管理员点击「确认审核」，**Then** 状态恢复为「已评估」，此时可执行「立即投标」或「放弃投标」
6. **Given** 投标管理员在已审核通过的标讯详情页，**When** 点击「立即投标」，**Then** 系统创建关联投标项目，状态变为「投标中」
6. **Given** 投标管理员在已评估标讯详情页，**When** 点击「放弃投标」并填写原因，**Then** 状态变为「已放弃」，原因记录在标讯字段

### User Story 2 - 转派跟踪中标讯到其他项目负责人 (Priority: P2)

投标管理员或投标组长需要把已分配但跟进不力或负责人变动的标讯转交给其他项目负责人。当前「分配」和「转派」混在同一个对话框，无法区分场景；PRD §4.2.6 要求转派是独立路径，且转派后原负责人立即失去访问权限。

**Why this priority**: 转派是日常运营高频操作（人员调整、负责人离职、优先级调整）。当前缺少独立流程导致团队只能通过「先取消分配再重新分配」绕过，留下双重操作日志且原负责人在中间状态仍能访问敏感评估信息，存在合规隐患。但绕过方案可用，因此不阻塞演示交付，定为 P2。

**Independent Test**: 投标管理员登录 → 列表页找到一条「跟踪中」标讯（项目负责人是 A） → 点击行操作中的「转派」按钮（与「分配」区分） → 弹窗仅展示「项目负责人」单选 → 选择 B → 确认 → 原负责人 A 刷新列表看不到该标讯 → 新负责人 B 列表中出现该标讯 → 状态保持「跟踪中」 → 操作日志记录原负责人 A 转为新负责人 B。

**Acceptance Scenarios**:

1. **Given** 跟踪中或已评估状态的标讯，**When** 投标管理员/组长查看列表，**Then** 操作列展示「转派」按钮（与「分配」分开）
2. **Given** 投标管理员点击「转派」，**When** 弹窗打开，**Then** 仅展示项目负责人单选下拉，不展示投标负责人字段
3. **Given** 选择新项目负责人后点击确认，**When** 提交成功，**Then** 标讯的项目负责人字段更新，项目部门按新负责人所在部门自动刷新，标讯状态保持不变
4. **Given** 转派完成，**When** 原项目负责人刷新列表/详情页，**Then** 该标讯立即不可见（数据权限即时生效）
5. **Given** 项目负责人或投标专员角色，**When** 查看「跟踪中」标讯，**Then** 不展示「转派」按钮
6. **Given** 标讯状态为「投标中」「已中标」「未中标」「已放弃」，**When** 任何角色查看列表，**Then** 不展示「转派」按钮
7. **Given** 投标管理员选择当前项目负责人作为转派目标，**When** 点击确认，**Then** 系统提示「不能转派给当前负责人」

### User Story 3 - 投标管理员配置标讯源并团队共享 (Priority: P3)

投标管理员通过「标讯源配置」对话框设置第三方平台拉取参数（平台多选、API 端点、关键字、地区、预算范围等）。当前配置存在浏览器本地，换浏览器或换电脑就丢失，团队成员看到的配置不一致，定时同步任务无法读取统一配置。

**Why this priority**: PRD §4.2.3 明确要求标讯源配置由投标管理员统一管理。当前 localStorage 方案在单人本地演示可用，但实际多人团队场景下，定时拉取任务和其他用户都看不到配置，影响标讯自动采集流程闭环。不阻塞 P1 评估表的核心闭环，因此定为 P3。

**Independent Test**: 投标管理员在浏览器 A 登录 → 打开配置弹窗 → 填写平台+API+关键字+地区+预算+自动同步 → 点击保存 → 退出 → 用浏览器 B 重新登录同账号 → 打开配置弹窗 → 看到刚才保存的所有配置值。同时以非管理员角色访问 → 后端返回 403，前端入口对其他角色隐藏。

**Acceptance Scenarios**:

1. **Given** 投标管理员身份，**When** 打开标讯源配置弹窗，**Then** 系统从后端拉取最新配置自动填充表单
2. **Given** 投标管理员修改配置后点击保存，**When** 表单校验通过，**Then** 配置写入数据库并 Toast 提示「配置保存成功」
3. **Given** 投标管理员保存配置后，**When** 另一位投标管理员或定时任务读取配置，**Then** 看到的是同一份最新配置
4. **Given** 投标组长/项目负责人/投标专员角色，**When** 列表页查看顶部操作区，**Then** 不展示「标讯源配置」按钮（入口隐藏）
5. **Given** 非投标管理员角色通过 API 直接访问配置接口，**When** 发起请求，**Then** 系统返回 403 拒绝
6. **Given** 投标管理员勾选「第三方商机服务」但未填 API 端点/密钥，**When** 点击保存，**Then** 表单校验拦截并提示具体缺失项
7. **Given** API 密钥在数据库中存储，**When** 通过任何方式查看数据库，**Then** 密钥以加密形式存储，不可明文读取

### User Story 4 - 按四种来源精确统计与筛选标讯 (Priority: P4)

业务侧需要按「第三方平台/CRM 商机/人工录入/批量导入」四种来源分别统计标讯数量并支持筛选。当前系统只区分「MANUAL/EXTERNAL」两种来源，导致 CRM 转入和第三方拉取混在 EXTERNAL，人工单条录入和批量 Excel 导入混在 MANUAL，运营无法分析各渠道贡献。

**Why this priority**: 数据完整性问题，影响投标管理员对各采集渠道的运营决策（哪个平台 ROI 高、批量导入占比、CRM 转化率等）。但当前 MANUAL/EXTERNAL 二分类不阻塞业务流程，所有标讯都能正确进入状态流转，因此定为 P4。

**Independent Test**: 通过 4 个不同来源各创建 1 条标讯（人工单条/批量上传 Excel/CRM 接口/第三方平台拉取） → 在列表筛选器选择「来源平台」 → 看到 4 个独立选项 → 分别筛选每个来源 → 每条标讯出现在正确来源类目下 → 调用统计接口看到按 4 个来源分组的数量。

**Acceptance Scenarios**:

1. **Given** 系统有 4 种来源的历史标讯，**When** 用户打开列表筛选器，**Then** 来源筛选下拉展示 4 个独立选项（第三方平台 / CRM 商机 / 人工录入 / 批量导入）
2. **Given** 通过人工单条录入创建标讯，**When** 写入数据库，**Then** source_type 字段标识为「人工录入」类目
3. **Given** 通过 Excel 批量导入创建标讯，**When** 写入数据库，**Then** source_type 字段标识为「批量导入」类目
4. **Given** 通过 CRM 接口创建标讯，**When** 写入数据库，**Then** source_type 字段标识为「CRM 商机」类目
5. **Given** 通过第三方平台拉取创建标讯，**When** 写入数据库，**Then** source_type 字段标识为「第三方平台」类目
6. **Given** 调用统计接口，**When** 按来源分组，**Then** 返回 4 个来源各自的标讯计数
7. **Given** 历史数据迁移完成，**When** 查询任何标讯，**Then** 不存在旧 source_type 值（MANUAL/EXTERNAL）的记录

### User Story 5 - 业务术语「项目负责人」统一显示 (Priority: P5)

业务侧（PRD/培训材料/UI）统一使用「项目负责人」描述跟进标讯评估的角色，但当前系统在多处显示「销售」「业务负责人」等不一致表述，导致新用户混淆角色定位、培训文档对不上 UI 实际显示。

**Why this priority**: 仅术语显示一致性问题，不影响任何功能逻辑、权限或数据。因此优先级最低。但与 P1-P4 同期上线有助于一次性收敛 PRD 与代码的术语差异。

**Independent Test**: 用销售角色用户登录 → 查看左上角个人信息 → 显示「项目负责人」而非「销售」 → 在分配弹窗选择项目负责人下拉 → 选项标签显示「项目负责人」 → 在标讯详情页查看项目负责人字段 → 文案统一为「项目负责人」 → 在系统设置查看角色列表 → 该角色描述为「项目负责人」。

**Acceptance Scenarios**:

1. **Given** 销售（sales）角色用户登录，**When** 系统任何界面显示其角色，**Then** 显示文本为「项目负责人」
2. **Given** 任何角色查看分配/转派弹窗的项目负责人下拉，**When** 渲染选项，**Then** 标签使用「项目负责人」术语
3. **Given** 数据库中现有 sales 角色用户，**When** 查询用户信息，**Then** 角色代码不变，仅展示名称变化
4. **Given** 现有权限矩阵和数据可见范围规则，**When** 角色重命名后，**Then** 所有权限规则和数据可见范围保持不变

---

### Edge Cases

- 评估表的客户信息 13×14 表格中，若某些行（例如「专家 3」）暂未确定，单元格允许留空，但提交时所有「触达方式」必填字段不能为空
- 评估表已评估状态下再编辑保存后，状态变为「已评估-待审核」。投标管理员审核期间，标讯底部的「立即投标」「放弃投标」按钮不可用
- 转派时新负责人账号已停用/不存在：系统应在弹窗下拉中过滤掉停用账号，避免选择
- 转派的同一时刻原负责人正在编辑评估表草稿：草稿应在转派后保留给新负责人，并提示「上一位负责人有草稿」
- TenderSource 配置保存时 API 端点拼写错误或网络不通：测试连接按钮应能区分「未点击测试」「测试失败」「测试成功」三种状态
- 历史标讯 source_type 数据回填规则不能匹配明确来源时（脏数据），统一回填为「人工录入」并打 `legacy_unknown` 标记，便于后续清理
- 角色重命名后，旧版本的国际化文案缓存：用户首次刷新可能仍看到旧术语，需在前端构建时统一替换

## Requirements *(mandatory)*

### Functional Requirements

#### 评估表三段式（FR-001 ~ FR-008）

- **FR-001**: 系统必须把标讯评估表组织为三段，按顺序展示：基础信息、客户信息、投标负责人建议
- **FR-002**: 基础信息段必须包含至少 8 个字段：入围家数（数值）、年度电商采购金额（数值，万元）、招标文件不利项（文本）、风险预判举例（文本）、风险兜底方案（文本）、项目经理是否了解评标全流程（文本）、需要的支持及关键信息备注（文本）、项目计划 GAP（文本）
- **FR-003**: 客户信息段必须呈现为固定 13 行 × 14 列的表格，不允许用户增删行
- **FR-004**: 客户信息表格的 13 行必须对应固定 13 个角色：项目最高决策人、物资公司董事长、物资公司分管电商领导、电商公司董事长、电商公司总经理、电商公司副总经理、电商公司运营负责人、招标文件制作人、其他关键决策人 1/2/3、专家 1/2/3
- **FR-005**: 客户信息表格的 14 列必须对应固定 14 个信息维度，字段类型按 PRD 原文定义：**固定文本不可编辑**（客户信息（角色名））；**自由文本单行**（姓名、职位、西域项目负责人、触达方式、倾向性评估依据）；**下拉选择（是/否）**（是否触达、是否有正式高层交流、是否向此人引导标书、是否可获取关键信息、是否可删除不利项、是否为重点攻克对象、是否可同步评标信息）；**下拉选择（支持/中立/反对）**（对我司的倾向性）。角色名和姓名支持模糊匹配搜索。
- **FR-006**: 投标负责人建议段必须包含「是否投标」（是/否）+ 「理由」（文本，选「否」时必填）
- **FR-007**: 系统必须在评估表提交时校验三段必填项完整性，缺失时拒绝提交并标注具体缺失字段
- **FR-008**: 系统必须支持「已评估」状态下重新编辑保存评估表，保存后状态变为「已评估-待审核」（`EVALUATED_PENDING_REVIEW`），原数据被覆盖，操作日志记录变更前后摘要，并向投标管理员/组长发送重新审核待办通知，待审核确认后方可恢复「已评估」并允许决策

#### 转派流程（FR-009 ~ FR-014）

- **FR-009**: 系统必须为「跟踪中」「已评估」状态的标讯提供独立的「转派」入口，与「分配」按钮在 UI 上明确区分
- **FR-010**: 转派功能必须只对「投标管理员」「投标组长」角色可见和可执行，其他角色 403 拒绝
- **FR-011**: 转派对话框必须仅展示「项目负责人」单选字段，不包含投标负责人字段
- **FR-012**: 转派完成后系统必须更新标讯的项目负责人字段、项目部门字段（按新负责人所在部门自动带出），状态保持不变
- **FR-013**: 转派完成后原项目负责人必须立即失去对该标讯的查看权限（数据权限即时生效，无需等待会话刷新）
- **FR-014**: 系统必须拒绝把标讯转派给当前项目负责人本身，并提示「不能转派给当前负责人」

#### 标讯源后端持久化（FR-015 ~ FR-020）

- **FR-015**: 系统必须把标讯源配置存储在后端数据库，整个团队共享一份生效配置
- **FR-016**: 系统必须仅允许「投标管理员」角色读写标讯源配置，其他角色访问相应 API 时返回 403
- **FR-017**: 前端「标讯源配置」按钮必须仅对投标管理员角色可见，其他角色入口隐藏
- **FR-018**: 系统必须在数据库中以加密形式存储 API 密钥字段，不可明文读取
- **FR-019**: 系统必须保留现有的「测试连接」能力，并能区分「未测试」「测试中」「测试成功」「测试失败」四种状态
- **FR-020**: 配置变更必须写入审计日志，记录操作人、时间、变更前后值（密钥字段脱敏）

#### 来源类型扩展（FR-021 ~ FR-024）

- **FR-021**: 系统必须把标讯来源类型从 2 个值扩展为 4 个值，分别对应：第三方平台、CRM 商机、人工录入、批量导入
- **FR-022**: 系统必须在标讯创建时根据创建入口正确写入 source_type 字段，4 种来源各自映射明确的枚举值
- **FR-023**: 列表筛选和统计接口必须支持按 4 个独立来源筛选与分组
- **FR-024**: 历史数据迁移完成后，数据库中不应存在 source_type 为旧值（MANUAL/EXTERNAL）的记录

#### 角色术语统一（FR-025 ~ FR-027）

- **FR-025**: 系统必须把销售（sales）RoleProfile 的中文展示名统一为「项目负责人」
- **FR-026**: 系统必须保持销售（sales）的 RoleProfile 代码不变，现有权限和数据可见范围不变
- **FR-027**: 所有 UI 显示「销售」「业务负责人」等旧术语的位置必须统一替换为「项目负责人」（包括但不限于：角色徽章、分配下拉选项、详情页字段名、左侧导航、登录提示）

### Key Entities *(include if feature involves data)*

- **TenderEvaluation**: 单条标讯的评估表聚合，包含基础信息段（8 字段）、客户信息段（13×14 矩阵）、投标负责人建议段（2 字段）、提交状态、审核字段；与 Tender 一对一关系
- **TenderEvaluationCustomerInfo**: 客户信息段的单元格数据，主键为评估 ID + 角色键 + 信息键三元组；role_key 取自 13 个角色枚举；info_key 取自 14 个信息维度枚举
- **TenderAssignmentRecord**: 标讯分配历史，新增 type 字段区分「分配」和「转派」；记录 from_owner / to_owner / operator / occurred_at
- **TenderSourceConfig**: 全表单例的标讯源配置聚合，包含平台列表、API 端点、加密 API 密钥、关键字、地区列表、预算上下限、自动同步开关与频率、自动去重开关；含 updated_by 字段标识最后修改者
- **Tender 来源类型枚举**: 从 2 值扩展为 4 值，对应第三方平台、CRM 商机、人工录入、批量导入四种来源

## Clarifications

### Session 2026-05-18

- Q: 评估表再编辑是否需要重新审核？→ A: **严格审核（Option A）**。项目负责人重新编辑评估表保存后，状态变为「已评估-待审核」（`EVALUATED_PENDING_REVIEW`），投标管理员必须显式点击「确认审核」按钮后才能恢复「已评估」状态、允许执行「立即投标/放弃投标」决策。每次编辑都触发一次新的审核流程，以投标管理员最新审核结论为准。
- Q: 客户信息 13×14 表格的字段类型？→ A: **PRD 原文为准**。客户信息（角色名）为固定展示文本不可编辑；姓名、职位、西域项目负责人、触达方式、倾向性评估依据为自由文本（单行输入）；是否触达（是/否）、对我司的倾向性（支持/中立/反对）、是否有正式的高层交流（是/否）、是否向此人引导标书（是/否）、是否可获取关键信息（是/否）、是否可删除不利项（是/否）、是否为重点攻克对象（是/否）、是否可同步评标信息（是/否）为下拉选择（单选，非自由文本）。
- Q: 历史 EXTERNAL 标讯如何区分来源？→ A: **按 source 字段前缀匹配（Option B）**。`SELECT DISTINCT source FROM tenders WHERE source_type='EXTERNAL'` 确定可识别的 CRM 前缀模式，`source` 以 `CRM` 或 `商机` 开头的迁为 `CRM_OPPORTUNITY`，其余迁为 `EXTERNAL_PLATFORM`。先跑预览脚本确认分布再执行迁移。

## Assumptions

- 客户信息 13 个角色与 14 个信息维度是 PRD 当前版本（V1.1）的最终方案，后续若调整需要走专门的需求变更流程
- 评估表「已评估」状态下的重新编辑：采用覆盖原数据 + 操作日志记录变更，不做版本化保存
- 转派后原负责人的实时权限失效依赖现有数据权限拦截器和项目访问域服务即时计算，不引入会话广播或主动 push 机制
- 标讯源配置全表单例：整个组织共享一份生效配置，不区分团队/部门级别；如未来需要按部门拆分，作为独立需求
- API 密钥加密采用现有已部署的对称加密方案，不引入新的密钥管理基础设施
- 历史 source_type 数据回填规则：先跑 `SELECT DISTINCT source FROM tenders WHERE source_type='EXTERNAL'` 预览数据分布；`source` 以 `CRM` 或 `商机` 开头的迁为 `CRM_OPPORTUNITY`；能识别批量导入的回填为 `BULK_IMPORT`；其余 `EXTERNAL` 迁为 `EXTERNAL_PLATFORM`；其余 `MANUAL` 迁为 `MANUAL_SINGLE`；存在歧义的统一回填为 `MANUAL_SINGLE` 并打 `legacy_unknown` 标记
- 角色重命名仅修改展示层，不修改 RoleProfile 代码、不修改权限矩阵、不修改数据可见范围规则，因此不需要数据迁移
- 本 P0 范围内不涉及 CRM 商机入站 webhook 接口的实现（CRM 推送商机数据自动建标讯的能力）；该能力作为独立 P1 需求另行规划
- 本 P0 范围内不补齐标讯列表的 18 列（当前 16 列）和标讯源配置的 10 字段（当前 8 字段）；剩余 2 列/2 字段待 PRD 二次确认后另行规划
- AI 解析能力（PDF/Word 自动建标讯）沿用现有文档解析链路，不在本 P0 范围内扩展
