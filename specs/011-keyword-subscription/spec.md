# Feature Specification: 标讯关键词订阅

**Feature Branch**: `011-keyword-subscription`

**Created**: 2026-05-27

**Status**: Draft

**Input**: 标讯关键词订阅功能。用户可以设置关键词组合（多个关键词 AND/OR 关系），系统每天自动匹配新增标讯，匹配结果通过站内通知推送给用户。先做后端 API 和数据库设计，前端只做基础页面。

## User Scenarios & Testing

### User Story 1 - 管理关键词订阅规则 (Priority: P1)

用户需要能够创建和管理标讯关键词订阅规则。用户可以设置一组关键词，并指定它们之间的逻辑关系（AND/OR），以精确匹配感兴趣的标讯。

**Why this priority**: 关键词订阅规则管理是所有后续功能（匹配、通知）的基础，没有规则就无法进行匹配。

**Independent Test**: 用户登录系统后，可以进入关键词订阅管理页面，创建一条包含多个关键词的订阅规则，并可查看、编辑和删除已有规则。

**Acceptance Scenarios**:

1. **Given** 用户已登录系统，**When** 用户创建一条关键词订阅规则，填写规则名称、选择关键词逻辑关系（AND），输入关键词"建筑工程"和"上海"，**Then** 系统成功保存该规则，并在规则列表中显示。
2. **Given** 用户有一条已保存的关键词订阅规则，**When** 用户编辑该规则，修改关键词或逻辑关系，**Then** 系统成功更新该规则。
3. **Given** 用户有一条已保存的关键词订阅规则，**When** 用户删除该规则，**Then** 系统成功删除该规则，不再对该规则执行匹配。

---

### User Story 2 - 接收标讯匹配结果 (Priority: P1)

系统每天自动运行匹配任务，将新增标讯与用户的所有订阅规则进行匹配，匹配结果通过站内通知推送给用户。

**Why this priority**: 这是关键词订阅的核心价值——自动发现用户感兴趣的标讯并及时通知。

**Independent Test**: 管理员可以手动触发一次匹配任务，系统执行匹配后，用户的站内通知列表中会出现匹配结果的提醒。

**Acceptance Scenarios**:

1. **Given** 用户有已保存的关键词订阅规则，**When** 系统执行每日匹配任务，发现新增标讯匹配该规则，**Then** 系统生成一条站内通知推送给该用户。
2. **Given** 用户收到标讯匹配通知，**When** 用户点击通知，**Then** 系统跳转到匹配结果列表页面，展示本次匹配到的具体标讯。
3. **Given** 用户有多条订阅规则匹配到标讯，**When** 系统推送通知，**Then** 用户可以查看每条规则各自的匹配结果。

---

### User Story 3 - 查看匹配结果 (Priority: P2)

用户可以查看每条订阅规则的匹配历史，了解哪些标讯被匹配到，以及匹配的时间和数量。

**Why this priority**: 提供匹配结果的追溯能力，帮助用户评估订阅规则的效果。

**Independent Test**: 用户进入某条订阅规则的详情页面，可以看到该规则的匹配历史记录，包括每次匹配的时间、匹配到的标讯数量。

**Acceptance Scenarios**:

1. **Given** 用户有一条订阅规则已有匹配记录，**When** 用户查看该规则的匹配历史，**Then** 系统展示每条匹配记录的时间戳、匹配到的标讯列表。
2. **Given** 用户查看匹配结果列表，**When** 用户点击某条标讯，**Then** 系统跳转到标讯详情页面（复用现有标讯详情功能）。

---

### Edge Cases

- 关键词为空时创建规则：系统应提示用户至少输入一个关键词。
- 关键词包含特殊字符：系统应对关键词进行合理的清理和转义处理。
- 长时间无匹配结果：系统应在规则列表中提示用户该规则当前无匹配结果。
- 大量关键词（超过10个）：系统应提示用户关键词数量上限。
- 用户取消订阅规则：取消后不再产生新的匹配和通知，但历史匹配结果保留。
- 新增标讯同时匹配多条规则：每条匹配规则都应产生独立的匹配记录和通知。

## Requirements

### Functional Requirements

- **FR-001**: 系统 MUST 支持用户创建关键词订阅规则，每条规则包含规则名称、关键词列表和逻辑关系类型。
- **FR-002**: 系统 MUST 支持关键词之间的 AND（全部匹配）和 OR（任一匹配）两种逻辑关系。
- **FR-003**: 系统 MUST 支持用户编辑和删除已创建的订阅规则。
- **FR-004**: 系统 MUST 至少每天执行一次自动匹配任务（可通过调度任务配置），将新增标讯与所有活跃订阅规则进行匹配。
- **FR-005**: 系统 MUST 在匹配成功后向该订阅规则所属用户发送站内通知。
- **FR-006**: 系统 MUST 记录每次匹配的结果，包括匹配时间、匹配到的标讯列表。
- **FR-007**: 系统 MUST 支持用户查看每条订阅规则的匹配历史记录。
- **FR-008**: 系统 MUST 支持用户在匹配结果列表中点击标讯跳转到标讯详情页。
- **FR-009**: 系统 MUST 在关键词数量超过上限（默认10个）时提示用户。
- **FR-010**: 系统 MUST 支持用户启用/停用订阅规则，停用后不再执行匹配。
- **FR-011**: 系统 MUST 支持用户查询自己的所有订阅规则列表。
- **FR-012**: 系统 MUST 保留匹配历史记录，供用户随时查阅。

### Key Entities

- **KeywordSubscription**（关键词订阅规则）: 代表用户创建的一条订阅规则。包含用户ID、规则名称、关键词列表、逻辑关系类型（AND/OR）、启用状态。与 User（用户）是多对一关系，与 KeywordMatchResult（匹配结果）是一对多关系。
- **KeywordSubscriptionKeyword**（订阅关键词）: 代表订阅规则中的单个关键词。包含关键词文本、关联的订阅规则ID。与 KeywordSubscription（订阅规则）是多对一关系。
- **KeywordMatchResult**（匹配结果）: 代表一次自动匹配的执行记录。包含关联的订阅规则ID、匹配执行时间、本次匹配到的标讯数量。与 KeywordSubscription（订阅规则）是多对一关系。
- **KeywordMatchItem**（匹配结果明细）: 代表匹配结果中的一条具体标讯。包含匹配结果ID、标讯ID（关联现有标讯）、标讯来源表名、标讯标题。与 KeywordMatchResult（匹配结果）是多对一关系，与 TenderNotice（标讯）是多对一关系。
- **Notification**（站内通知）: 代表发送给用户的站内消息。包含接收用户ID、通知类型、通知内容、相关匹配结果ID、已读状态。复用或扩展现有的通知系统。

## Success Criteria

### Measurable Outcomes

- **SC-001**: 用户可以创建包含至少10个关键词的订阅规则并成功保存。
- **SC-002**: 系统每日定时匹配任务在5分钟内处理完当日新增标讯与所有活跃规则的匹配。
- **SC-003**: 用户从收到通知到查看匹配结果的操作路径不超过3次点击。
- **SC-004**: 匹配结果推送的延迟不超过匹配任务完成后的5分钟。

## Assumptions

- 系统已有站内通知模块或基础设施，本功能将复用它，而非重新实现通知系统。
- 标讯数据已存在于系统中（通过现有的标讯采集/创建流程），本功能不涉及标讯数据的采集。
- 关键词匹配基于标讯标题和/或内容进行模糊匹配（具体匹配策略在技术设计中确定）。
- 匹配任务的执行时间默认为每天凌晨2:00（可配置）。
- 用户需要先登录系统才能使用订阅功能。
- 匹配历史数据不需要永久保留，默认保留最近90天。
